# DRF + 로컬 LLM으로 AI 핑계 생성기 만들기(1) : AI 핑계 생성기를 커뮤니티형 서비스로 확장하기

이번 교안은 **기존에 만든 `AI 핑계 생성기(1)` 교안의 확장편**입니다.
즉, 완전히 새로운 프로젝트를 처음부터 만드는 것이 아니라,
이미 만들어 둔 **LLM 기반 핑계 생성 서비스**를 실제 서비스처럼 보이도록 점진적으로 업그레이드합니다.

이번 확장편의 핵심은 아래 5가지입니다.

1. 기존 CRUD를 다시 정리하면서 **Generic 기반 구조를 더 명확하게 이해하기**
2. 여러 APIView / GenericAPIView 구조를 **ViewSet + Router 구조로 전환하기**
3. 설정값을 코드에 하드코딩하지 않고 **`.env` 기반 환경설정으로 분리하기**
4. 목록 API가 많아질 것을 고려해 **페이지네이션 추가하기**
5. 생성된 핑계에 썸네일이나 상황 이미지를 붙일 수 있도록 **이미지 업로드 기능 추가하기**

이번 교안은 특히 아래와 같은 사람에게 좋습니다.

* 1편에서 DRF + Local LLM 기본 흐름을 이미 한 번 실습한 사람 (<a href="https://website-git-new-landing-page-schick.vercel.app/docs/reference/source-files/define-document-type" target="_blank">이전 교안 보러가기</a>)

* CRUD는 이해했지만 `ViewSet`이 왜 필요한지 아직 감이 덜 오는 사람
* 실제 서비스처럼 설정을 분리하고 싶은 사람
* 단순 텍스트 API를 넘어 이미지 업로드까지 붙여 보고 싶은 사람

---

# 0. 이번 교안의 위치와 목표

기존 1편에서는 아래 흐름을 만들었습니다.

```
사용자 입력 (situation, tone)
→ DRF API 요청
→ 로컬 LLM(Qwen) 호출
→ DB 저장
→ 목록 / 상세 / 랜덤 조회
→ HTML 페이지에서 확인
```

이번 2편 확장 1에서는 이 구조를 아래처럼 발전시킵니다.

```
사용자 입력 (situation, tone, image)
→ Serializer 검증
→ ViewSet 라우팅
→ LLM 핑계 생성
→ DB 저장
→ 페이지네이션 목록 조회
→ 이미지 포함 HTML / API 응답 확인
→ .env 기반 설정 분리
```

즉, 핵심은 단순히 기능을 더 추가하는 것이 아니라,
**기존 코드를 더 서비스다운 구조로 재정비하는 것**입니다.

---

# 1. 이번 실습에서 배우는 것

이번 확장 실습을 마치면 아래 내용을 이해할 수 있습니다.

* 기존 `ListAPIView`, `RetrieveUpdateDestroyAPIView`, `GenericAPIView` 구조를 어떻게 `ViewSet`으로 묶는지
* `router.register()`가 왜 편한지
* API가 많아질 때 URL을 어떻게 더 깔끔하게 관리하는지
* `.env` 파일로 설정을 분리하는 이유
* 목록 API에 페이지네이션이 왜 필요한지
* `ImageField`를 사용할 때 `MEDIA_URL`, `MEDIA_ROOT`가 왜 필요한지
* 기존 HTML 테스트 페이지를 이미지 포함 구조로 어떻게 확장하는지

이번 확장 실습의 큰 키워드는 아래 3개입니다.

```
정리
확장
서비스화
```

즉, 이번 교안은 **"코드가 돌아간다"에서 끝나는 것이 아니라**
**"코드 구조가 왜 이렇게 정리되는지"**까지 이해하는 것이 중요합니다.

# 2. 이번 교안에서 무엇이 달라지나요?

기존 1편에서는 아래와 같은 구조였습니다.

| 구분       | 기존 구조                          |
| -------- | ------------------------------ |
| 목록 조회    | `ListAPIView`                  |
| 상세/수정/삭제 | `RetrieveUpdateDestroyAPIView` |
| 랜덤 조회    | `GenericAPIView`               |
| AI 생성    | `GenericAPIView`               |
| 설정       | `settings.py`에 직접 작성           |
| 목록 응답    | 전체 목록 한 번에 반환                  |
| 이미지      | 없음                             |

이번 2편 확장 1에서는 아래처럼 바뀝니다.

| 구분          | 확장 구조                   |
| ----------- | ----------------------- |
| 목록/상세/수정/삭제 | `ModelViewSet`으로 통합     |
| 랜덤 조회       | ViewSet의 `@action`으로 이동 |
| AI 생성       | ViewSet의 `@action`으로 이동 |
| 설정          | `.env` 파일 사용            |
| 목록 응답       | 페이지네이션 적용               |
| 이미지         | `ImageField` 추가         |

이 말은 곧,
기존의 기능은 사라지는 것이 아니라 **더 정리된 구조로 재배치되는 것**입니다.

# 5. 기존 코드에서 무엇을 수정하는가?

# 8. `.env` 환경설정 추가

`settings.py`에 중요한 값은 항상 직접 작성하지 않고 .env에 작성하도록 합니다.

만약 프로젝트가 커지고 별도의 .env를 작성하지 않는다면 아래와 같은 문제가 생깁니다.

* 민감한 값이 코드에 그대로 남는다.
* 개발/배포 환경을 나누기 어렵다.
* 협업 시 개인 환경값이 충돌한다.

그래서 이제 `.env`를 사용합니다.

## 8-1. 프로젝트 루트에 `.env` 파일 생성

```env title=".env"
SECRET_KEY=django-insecure-change-this-secret-key
DEBUG=True
ALLOWED_HOSTS=127.0.0.1,localhost
LLM_MODEL=Qwen/Qwen2.5-1.5B-Instruct
PAGE_SIZE=5
```

## 값 설명

| 키               | 설명              |
| --------------- | --------------- |
| `SECRET_KEY`    | Django 보안 키     |
| `DEBUG`         | 개발 환경 여부        |
| `ALLOWED_HOSTS` | 허용할 호스트 목록      |
| `LLM_MODEL`     | 사용할 허깅페이스 모델 이름 |
| `PAGE_SIZE`     | 목록 한 페이지당 개수    |

## 9-2. 기존 코드에서 달라진 점

### 1. `from decouple import config, Csv`

이 부분이 새로 들어갑니다.
`.env` 값을 읽기 위해 필요합니다.

### 2. `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`

예전에는 문자열이나 리스트를 직접 적었지만,
이제 `.env`에서 읽습니다.

### 3. `MEDIA_URL`, `MEDIA_ROOT`

### 4. `REST_FRAMEWORK`

그러면 목록 API마다 페이지네이션 클래스를 반복 작성하지 않아도 됩니다.

### 5. `LLM_MODEL`

서비스 로직 안에서 모델명을 직접 적지 않고,
설정값을 읽게 만들 준비입니다.

# 10. models.py 수정 : 이미지 필드 추가

기존 1편의 `Excuse` 모델은 텍스트 중심이었습니다.
이번에는 상황 이미지를 첨부할 수 있도록 `image` 필드를 추가합니다.

## 10-1. 수정 후 전체 코드

```python title="excuses/models.py"
from django.db import models


class Excuse(models.Model):
    situation = models.CharField(max_length=100)
    tone = models.CharField(max_length=50)
    content = models.TextField()
    image = models.ImageField(upload_to='excuses/', blank=True, null=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    def __str__(self):
        return f"[{self.tone}] {self.situation}"
```

## 10-2. 기존 코드에서 어떻게 바뀌었나?

기존에는 아래 필드만 있었습니다.

```
situation
tone
content
created_at
updated_at
```

이번에는 여기에 아래가 추가됩니다.

```python
image = models.ImageField(upload_to='excuses/', blank=True, null=True)
```

## 10-3. 왜 `blank=True, null=True`를 쓰나요?

이번 프로젝트는 **이미지가 선택 기능**이기 때문입니다.
즉,

* 이미지 없이도 핑계 생성 가능
* 필요할 때만 이미지 업로드 가능

이 구조가 더 유연합니다.

## 10-4. `upload_to='excuses/'`

업로드된 파일은 `media/excuses/` 경로 아래 저장됩니다.

예를 들어 `meeting.png`를 업로드하면 대략 이런 식으로 저장됩니다.

```text
media/excuses/meeting.png
```

# 11. migrations 적용

모델을 바꿨으므로 마이그레이션이 필요합니다.

```bash
python manage.py makemigrations
python manage.py migrate
```

## 설명

* `makemigrations`: 모델 변경사항을 migration 파일로 생성
* `migrate`: 실제 DB 테이블에 반영

이미지를 추가했는데 migrate를 하지 않으면,
Django는 여전히 이미지가 추가되지 않은 이전 테이블 구조라고 생각하기 때문에 오류가 납니다.

# 12. serializers.py 수정

기존 1편에서는 `ExcuseSerializer`, `ExcuseGenerateSerializer` 두 개를 나눠서 사용했습니다.
이 구조는 이번에도 유지하는 것이 좋습니다.

왜냐하면,

* 목록/상세/수정 응답은 전체 모델 필드를 다뤄야 하고
* AI 생성 입력은 `situation`, `tone`, `image`만 받는 용도로 분리하는 편이 더 명확하기 때문입니다.

## 12-1. 전체 코드

```python title="excuses/serializers.py"
from rest_framework import serializers
from .models import Excuse


class ExcuseSerializer(serializers.ModelSerializer):
    image_url = serializers.SerializerMethodField()

    class Meta:
        model = Excuse
        fields = [
            'id',
            'situation',
            'tone',
            'content',
            'image',
            'image_url',
            'created_at',
            'updated_at',
        ]
        read_only_fields = ['id', 'created_at', 'updated_at']

    def get_image_url(self, obj):
        request = self.context.get('request')
        if obj.image and request:
            return request.build_absolute_uri(obj.image.url)
        if obj.image:
            return obj.image.url
        return None


class ExcuseGenerateSerializer(serializers.Serializer):

    tone = serializers.CharField(max_length=50)
    image = serializers.ImageField(required=False, allow_null=True)
```

# 13. services.py 수정

기존 1편에서는 서비스 로직 안에 모델명을 직접 적었을 가능성이 큽니다.
이번에는 `settings.LLM_MODEL`을 참조하도록 바꿉니다.

이렇게 하면 나중에 모델 이름을 바꾸고 싶을 때
서비스 함수 코드를 직접 수정하지 않아도 됩니다.

## 13-1. 전체 코드

```python title="excuses/services.py"
from django.conf import settings
from transformers import pipeline
import re

_generator = None


def get_generator():
    global _generator
    if _generator is None:
        _generator = pipeline(
            "text-generation",
            model=settings.LLM_MODEL,
        )
    return _generator


def clean_excuse_text(text: str) -> str:
    text = text.strip().strip('"').strip("'")

    # 첫 줄만 사용
    text = text.split("\n")[0].strip()

    # 해시태그 제거
    text = re.sub(r"#\S+", "", text)

    # 불필요한 말머리 제거
    text = text.replace("결과:", "").replace("핑계:", "").strip()

    # 2문장까지만 유지
    sentences = re.split(r"(?<=[.!?])\s+", text)
    text = " ".join(sentences[:2]).strip()

    # 이상한 꼬리 제거
    garbage_keywords = ["너무 긴 얘기", "설명", "해시태그", "출력", "작성", "예시"]
    for keyword in garbage_keywords:
        if keyword in text:
            text = text.split(keyword)[0].strip()

    # 너무 짧거나 비정상이면 fallback
    if len(text) < 8:
        return ""

    return text


def generate_excuse(situation: str, tone: str) -> str:
    generator = get_generator()

    prompt = f"""
너는 한국어 핑계를 만들어주는 도우미다.
반드시 아래 예시처럼 짧고 자연스럽게만 답해라.
설명, 해시태그, 따옴표, 말머리 없이 핑계 문장만 출력해라.

예시 1
상황: 지각
톤: 정중한
결과: 죄송합니다. 출근길 교통이 예상보다 많이 막혀서 조금 늦었습니다.

예시 2
상황: 과제 제출 지연
톤: 유머
결과: 거의 다 끝냈는데 마지막 정리 단계에서 자료가 생각보다 말을 안 듣네요. 조금만 더 다듬어서 바로 공유드릴게요.

상황: {situation}
톤: {tone}
결과:
""".strip()

    result = generator(
        prompt,
        max_new_tokens=40,
        do_sample=True,
        temperature=0.6,
        top_p=0.85,
        repetition_penalty=1.2,
        return_full_text=False,
    )

    content = result[0]["generated_text"].strip()
    content = clean_excuse_text(content)

    if not content:
        if tone == "유머":
            content = "거의 다 끝냈는데 마지막 정리 단계에서 자료가 저랑 밀당 중입니다. 조금만 더 다듬어서 바로 보여드릴게요."
        elif tone == "정중한":
            content = "죄송합니다. 발표 자료 마무리가 예상보다 길어져서 조금만 더 정리한 뒤 공유드리겠습니다."
        else:
            content = "발표 자료 마무리가 조금 지연되고 있어 잠시만 더 시간이 필요합니다."

    return content
```

# 14. 핵심 변경 : 기존 View들을 ViewSet으로 전환

이제 이번 교안의 핵심입니다.
기존에는 기능별로 뷰를 나눴습니다.

예를 들면 대략 이런 구조였을 것입니다.


이 방식은 입문용으로는 매우 좋습니다.
하지만 기능이 늘어나면 뷰 클래스 수가 많아지고,
URL도 점점 복잡해질 수 있습니다.

그래서 이번에는 **하나의 자원(resource) = 하나의 ViewSet** 구조로 정리합니다.

즉,

```
Excuse 관련 기능
→ ExcuseViewSet 하나에 모으기
```

# 15. views.py 전체 작성

## 15-1. 전체 코드

```python title="excuses/views.py"
import random

from django.shortcuts import render
from rest_framework import status, viewsets
from rest_framework.decorators import action
from rest_framework.parsers import MultiPartParser, FormParser, JSONParser
from rest_framework.response import Response

from .models import Excuse
from .serializers import ExcuseSerializer, ExcuseGenerateSerializer
from .services import generate_excuse


# HTML PAGE VIEWS

def home(request):
    return render(request, 'excuses/home.html')


def excuse_list_page(request):
    return render(request, 'excuses/excuse_list.html')


def excuse_detail_page(request, excuse_id):
    return render(request, 'excuses/excuse_detail.html', {'excuse_id': excuse_id})


def excuse_generate_page(request):
    return render(request, 'excuses/excuse_generate.html')


# API VIEWSET
class ExcuseViewSet(viewsets.ModelViewSet):
    queryset = Excuse.objects.all().order_by('-id')
    serializer_class = ExcuseSerializer
    parser_classes = [MultiPartParser, FormParser, JSONParser]

    def get_serializer_context(self):
        context = super().get_serializer_context()
        context['request'] = self.request
        return context

    @action(detail=False, methods=['get'], url_path='random')
    def random_excuse(self, request):
        excuses = list(self.get_queryset())

        if not excuses:
            return Response(
                {'message': '저장된 핑계가 없습니다.'},
                status=status.HTTP_200_OK,
            )

        excuse = random.choice(excuses)
        serializer = self.get_serializer(excuse)
        return Response(serializer.data)

    @action(
        detail=False,
        methods=['post'],
        url_path='generate',
        parser_classes=[MultiPartParser, FormParser, JSONParser],
    )
    def generate(self, request):
        serializer = ExcuseGenerateSerializer(data=request.data)
        serializer.is_valid(raise_exception=True)

        situation = serializer.validated_data['situation']
        tone = serializer.validated_data['tone']
        image = serializer.validated_data.get('image')

        content = generate_excuse(situation, tone)

        excuse = Excuse.objects.create(
            situation=situation,
            tone=tone,
            content=content,
            image=image,
        )

        result = self.get_serializer(excuse)
        return Response(result.data, status=status.HTTP_201_CREATED)
```

# 16. 왜 `ModelViewSet`을 쓰나요?

`ModelViewSet`은 아래 동작을 한 번에 묶어줍니다.

* 목록 조회 (`list`)
* 상세 조회 (`retrieve`)
* 생성 (`create`)
* 수정 (`update`, `partial_update`)
* 삭제 (`destroy`)

즉, 기존에 여러 개의 CRUD 뷰로 나누어 두었던 것을
**하나의 클래스에 일관성 있게 모아둘 수 있습니다.**

## 기존 방식과 비교

### 기존 방식

```
목록용 뷰
상세용 뷰
랜덤용 뷰
생성용 뷰
```

### ViewSet 방식

```
ExcuseViewSet
├─ list
├─ retrieve
├─ create
├─ update
├─ destroy
├─ random_excuse (@action)
└─ generate (@action)
```

이렇게 구조가 더 명확해집니다.

---

# 17. `@action`이 왜 중요한가요?

ViewSet의 기본 CRUD 외에 **추가 기능**이 필요할 때 사용합니다.
이번 프로젝트에서는 아래 2개가 대표적입니다.

* 랜덤 핑계 조회
* AI 핑계 생성

이 둘은 일반적인 CRUD 동작이 아닙니다.
그래서 `@action`을 사용합니다.

## 예시

```python
@action(detail=False, methods=['get'], url_path='random')
def random_excuse(self, request):
    ...
```

이렇게 작성하면 Router가 아래 URL을 자동으로 만들어 줍니다.

```
/api/excuses/random/
```


그리고 받은 배열을 바로 `forEach`로 돌렸을 가능성이 큽니다.

### 페이지네이션 적용 후

이제 응답 구조가 이렇게 바뀝니다.

```
{
  "count": 13,
  "next": "http://127.0.0.1:8000/api/excuses/?page=2",
  "previous": null,
  "results": [
    ...
  ]
}
```

즉,
이제는 `data`가 배열이 아니라 **페이지 정보 객체**이므로,
반드시 `data.results`를 순회해야 합니다.

이 부분이 이번 확장에서 가장 많이 헷갈리는 포인트 중 하나입니다.

# 26. excuse_generate.html 작성

이번 페이지는 이미지 업로드 때문에 기존보다 가장 많이 달라집니다.
기존에는 JSON POST만 보내면 되었지만,

```html title="excuses/templates/excuses/excuse_generate.html"

  <h2>AI 핑계 생성</h2>
  <p>상황과 톤을 입력하면 AI가 핑계를 생성하고 DB에 저장합니다. 이미지를 같이 올릴 수도 있습니다.</p>
</section>

<section class="card">
  <form id="generate-form" enctype="multipart/form-data">
    <label>상황</label>
    <input type="text" id="situation" required />

    <label>톤</label>
    <input type="text" id="tone" required />

    <label>상황 이미지 (선택)</label>

<script>
  document.getElementById('generate-form').addEventListener('submit', async (event) => {
    event.preventDefault();

    const formData = new FormData();
    formData.append('situation', document.getElementById('situation').value);
    formData.append('tone', document.getElementById('tone').value);

    const imageFile = document.getElementById('image').files[0];
    if (imageFile) {
      formData.append('image', imageFile);
    }

    const response = await fetch('/api/excuses/generate/', {
      method: 'POST',
      body: formData,
    });

    const data = await response.json();
    const box = document.getElementById('result-box');
    box.classList.remove('hidden');

    if (!response.ok) {
      box.innerHTML = `<p>생성 실패: ${JSON.stringify(data)}</p>`;
      return;
    }

    box.innerHTML = `
      <h3>생성 결과</h3>
      <p><strong>상황:</strong> ${data.situation}</p>
      <p><strong>톤:</strong> ${data.tone}</p>
      <p>${data.content}</p>
      ${data.image_url ? `<img src="${data.image_url}" alt="생성 이미지" class="preview-image" />` : ''}
      <p><a class="button-link" href="/excuses/${data.id}/page/">상세 페이지로 이동</a></p>
    `;
  });
</script>
{% endblock %}
```

## 왜 `FormData`를 써야 하나요?

기존 1편에서는 아래처럼 JSON 요청만 보내도 됐습니다.

```javascript
headers: { 'Content-Type': 'application/json' }
body: JSON.stringify({ situation, tone })
```

하지만 파일 업로드가 들어가면 JSON이 아니라 보통 `multipart/form-data` 형식이 필요합니다.
이때 브라우저에서 가장 간단히 쓰는 방법이 `FormData`입니다.

즉,

### 기존

텍스트만 전송

### 수정 후

텍스트 + 파일 함께 전송

이 차이 때문에 프론트 코드가 바뀝니다.

# 31. 이번 확장에서 자주 틀리는 포인트

## 1. `data.forEach is not a function`

### 원인

페이지네이션 적용 후에도 예전처럼 `data.forEach(...)`를 사용한 경우입니다.

### 해결

```javascript
data.results.forEach(...)
```

---

## 2. 이미지 업로드는 되었는데 화면에 안 보임

### 원인 후보

* `MEDIA_URL`, `MEDIA_ROOT` 누락
* `config/urls.py`의 `static()` 설정 누락
* serializer에서 `image_url` 처리 누락

---

## 3. `Cannot assign "<InMemoryUploadedFile...>": ...`

### 원인

업로드 파일은 받았는데 serializer/모델/파서 설정이 맞지 않는 경우입니다.

### 확인할 것

* 모델에 `ImageField`가 있는가
* `parser_classes`에 `MultiPartParser`, `FormParser`가 있는가
* 요청이 `FormData`로 가는가

---

## 4. 생성 페이지에서 400 오류가 남

### 원인

`FormData`를 보내면서 프론트에서 `Content-Type: application/json`을 수동 지정한 경우가 많습니다.

### 해결

`FormData`를 쓸 때는 보통 `headers`를 직접 지정하지 않습니다.
브라우저가 알아서 multipart boundary를 설정하게 두는 것이 좋습니다.

---

## 5. `.env`가 반영되지 않음

### 원인 후보

* 프로젝트 루트가 아닌 다른 위치에 `.env`를 둠
* `python-decouple` 미설치
* 서버 재시작 안 함

### 해결

* `.env` 위치 확인
* 패키지 설치 확인
* 서버 재실행

# 34. 오늘 배운 핵심 정리

## 1. 입문용 구조와 확장용 구조는 다를 수 있다.

기존 Generic 기반 구조는 학습용으로 매우 좋았습니다.
하지만 기능이 늘어나면 `ViewSet` 구조가 더 정리된 느낌을 줍니다.

## 2. `.env` 분리는 선택이 아니라 습관이다.

처음엔 귀찮아 보여도, 프로젝트가 커질수록 차이가 커집니다.

## 3. 페이지네이션을 넣으면 프론트 코드도 함께 바뀐다.

이 점을 꼭 기억해야 합니다.

## 4. 이미지 업로드는 모델, serializer, parser, urls, 템플릿이 모두 함께 맞아야 한다.

한 군데만 바꾸면 되는 기능이 아닙니다.

## 5. 기존 교안을 기반으로 확장하면 DRF 구조가 더 선명하게 보인다.

처음부터 복잡하게 만드는 것보다,
기존 프로젝트를 점진적으로 확장하는 방식이 훨씬 학습 효과가 좋습니다.
