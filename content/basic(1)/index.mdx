# Django 기초 - 공부시간 기록기 만들기(1) : 프로젝트 시작과 목록 페이지

이번 교안의 목표는 단순히 화면 하나를 띄우는 것이 아닙니다.

진짜 목표는 아래 흐름을 **처음부터 끝까지 손으로 직접 반복하면서** Django의 가장 기본 구조를 몸에 익히는 것입니다.

```
가상환경 생성
→ Django 설치
→ 프로젝트 생성
→ 앱 생성
→ settings.py 등록
→ 모델 작성
→ 마이그레이션
→ admin 등록
→ 관리자 페이지 확인
→ URL 연결
→ Template 작성
→ CSS 적용
→ 함수형 View로 목록 페이지 출력
```

Django를 처음 배우는 분들은 보통 아래 지점에서 많이 막힙니다.

* 프로젝트와 앱의 차이가 헷갈림
* settings.py는 언제 왜 수정해야 하는지 모름
* 모델을 만들고도 왜 DB에 안 생기는지 모름
* admin은 왜 쓰는지 체감이 안 됨
* URL, View, Template 관계가 머릿속에 안 잡힘

이번 교안은 이런 막힘을 줄이기 위해, **실습 순서를 실제 개발 흐름 기준으로 아주 천천히 재구성**했습니다.
즉 아직 만들지 않은 것을 먼저 쓰지 않고, 지금 단계에서 필요한 것만 차례대로 진행합니다.

# 0. 이번 교안의 진짜 목표

이번 교안은 "공부시간 기록기 서비스"를 만드는 교안이지만,
실제로 더 중요한 목표는 아래 3가지입니다.

| 목표            | 설명                                           |
| ------------- | -------------------------------------------- |
| Django 구조 익히기 | 프로젝트, 앱, 설정, 모델, URL, View, Template의 관계를 이해 |
| 데이터 흐름 익히기    | 모델을 만들고 DB에 반영하고 admin으로 확인하는 흐름 이해          |
| 화면 출력 익히기     | View에서 데이터를 조회하고 Template로 출력하는 흐름 이해        |

즉 이번 편에서는 아래 지점까지 가는 것이 핵심입니다.

```
DB에 저장된 공부 기록 데이터
→ Django View에서 조회
→ HTML Template로 전달
→ 예쁜 목록 페이지로 브라우저에 출력
```

이 흐름이 이해되면 다음 교안에서 만드는 Form / CRUD / Pagination / DRF도 훨씬 덜 어렵게 느껴집니다.

* View가 있어야 Template에 데이터를 넘길 수 있습니다.

즉 이번 교안은 **앞 단계가 다음 단계의 준비가 되도록 설계된 순서**입니다.


그래서 프로젝트 시작 전에는 항상 아래 흐름으로 확인하는 습관을 들이는 것이 좋습니다.

```
Python 확인
→ 가상환경 생성
→ 패키지 설치
→ 프로젝트 시작
```

# 5. 가상환경 만들기

Python 프로젝트에서는 **가상환경(virtual environment)** 을 사용하는 것이 매우 중요합니다.

## 가상환경이란?

가상환경은 현재 프로젝트만을 위한 독립된 Python 패키지 공간입니다.

쉽게 말하면,

```text
내 컴퓨터 전체 Python 환경과 분리된
이 프로젝트 전용 Python 공간
```

이라고 이해하면 됩니다.

## 왜 필요한가?

초보자는 처음에 이렇게 생각하기 쉽습니다.

> 그냥 pip install django 하면 되는 거 아닌가요?

물론 설치는 됩니다.
하지만 그렇게 하면 Django가 내 컴퓨터 전체 Python 환경에 들어갑니다.
그러면 다른 프로젝트와 버전이 섞이거나, 나중에 패키지 충돌이 생길 수 있습니다.

가상환경을 쓰면 아래처럼 프로젝트별로 독립적으로 관리할 수 있습니다.

```text
프로젝트 A
└─ Django 5.x

프로젝트 B
└─ FastAPI

프로젝트 C
└─ 데이터 분석용 패키지
```

즉 패키지가 섞이지 않습니다.

## 가상환경 생성

```bash title="terminal"
python -m venv .venv
```

환경에 따라 `python3`를 사용해야 할 수도 있습니다.

```bash title="terminal"
python3 -m venv .venv
```

## 가상환경 활성화

### Windows

```bash title="terminal"
.venv\Scripts\activate
```

### macOS / Linux

```bash title="terminal"
source .venv/bin/activate
```

활성화되면 보통 터미널 앞부분에 아래처럼 표시됩니다.

```text
(.venv)
```

## 지금 꼭 확인할 것

* `.venv` 폴더가 생성되었는지
* 터미널 앞에 `(.venv)`가 붙었는지

이 두 가지가 확인되면 정상입니다.

---

# 6. Django 설치

이제 가상환경 안에 Django를 설치합니다.

```bash title="terminal"
pip install django
```

설치가 끝나면 버전을 확인합니다.

```bash title="terminal"
django-admin --version
```

예시:

```text
5.2.1
```

## 왜 버전까지 확인할까?

패키지를 설치한 뒤에는 항상 아래를 확인하는 습관이 중요합니다.

* 정말 설치가 되었는지
* 지금 어떤 버전이 설치되었는지
* 가상환경 안에 잘 설치되었는지

실제로 초보자는 가상환경을 켜지 않은 상태에서 설치하거나,
아예 다른 Python 환경에 설치해서 혼란을 겪는 경우가 많습니다.

그래서 설치 후 버전 확인은 아주 좋은 습관입니다.

# 7. Django 프로젝트 생성

이제 Django 프로젝트 자체를 생성합니다.

```bash title="terminal"
django-admin startproject config .
```

## 이 명령어의 의미

초보자 입장에서는 이 명령어가 조금 낯설 수 있습니다.

```text
startproject config .
```

여기서:

* `config` 는 Django 설정 패키지 이름
* 마지막 `.` 은 “현재 폴더에 바로 생성하라”는 뜻

입니다.

즉 지금 작업 중인 `study_time_tracker` 폴더 안에 Django 프로젝트를 바로 생성하겠다는 의미입니다.

## 생성 후 구조

```text
study_time_tracker/
├── .venv/
├── config/
│   ├── __init__.py
│   ├── asgi.py
│   ├── settings.py
│   ├── urls.py
│   └── wsgi.py
└── manage.py
```

## 각 파일 설명

| 파일                 | 역할               |
| ------------------ | ---------------- |
| manage.py          | Django 명령 실행 진입점 |
| config/settings.py | 프로젝트 전체 설정 파일    |
| config/urls.py     | 최상위 URL 연결 파일    |
| config/wsgi.py     | 배포용 서버 연결        |
| config/asgi.py     | 비동기 서버 연결        |

이번 교안에서는 특히 아래 두 파일이 가장 중요합니다.

* `config/settings.py`
* `config/urls.py`

즉, 본격 실습 전에 **최소 실행 환경 검증**을 하는 것입니다.

# 9. 앱 생성

Django에서는 기능 단위를 **앱(App)** 으로 나누어 관리합니다.

이번 실습에서는 공부 기록 기능을 담당할 `study` 앱을 생성합니다.

```bash title="terminal"
python manage.py startapp study
```

생성 후 구조는 아래와 비슷합니다.

```text
study/
├── __init__.py
├── admin.py
├── apps.py
├── migrations/
├── models.py
├── tests.py
└── views.py
```

## 프로젝트와 앱의 차이

이 부분은 초보자가 많이 헷갈립니다.

### 프로젝트

프로젝트는 전체 웹서비스의 큰 틀입니다.

### 앱

앱은 프로젝트 안에서 특정 기능을 맡는 단위입니다.

예를 들어 큰 서비스라면:

* 사용자 앱
* 게시판 앱
* 주문 앱
* 결제 앱

처럼 나눌 수 있습니다.

이번 실습은 단순하므로 `study` 앱 하나만 사용하지만,
Django는 기본적으로 이렇게 **기능별 분리**를 지향합니다.

# 10. settings.py에 앱 등록

앱을 만들었다고 해서 Django가 자동으로 사용해주지는 않습니다.
반드시 `INSTALLED_APPS`에 등록해야 합니다.

그리고 이번 실습에서는 템플릿과 CSS도 사용할 것이므로,
해당 설정도 함께 넣어주겠습니다.

```python title="config/settings.py"
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent.parent

SECRET_KEY = 'django-insecure-change-this-key'

DEBUG = True

ALLOWED_HOSTS = []

INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',

    'study',
]

MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
]

ROOT_URLCONF = 'config.urls'

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [BASE_DIR / 'templates'],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
            ],
        },
    },
]

WSGI_APPLICATION = 'config.wsgi.application'

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': BASE_DIR / 'db.sqlite3',
    }
}

AUTH_PASSWORD_VALIDATORS = [
    {
        'NAME': 'django.contrib.auth.password_validation.UserAttributeSimilarityValidator',
    },
    {
        'NAME': 'django.contrib.auth.password_validation.MinimumLengthValidator',
    },
    {
        'NAME': 'django.contrib.auth.password_validation.CommonPasswordValidator',
    },
    {
        'NAME': 'django.contrib.auth.password_validation.NumericPasswordValidator',
    },
]

LANGUAGE_CODE = 'ko-kr'
TIME_ZONE = 'Asia/Seoul'
USE_I18N = True
USE_TZ = True

STATIC_URL = '/static/'
STATICFILES_DIRS = [BASE_DIR / 'static']

DEFAULT_AUTO_FIELD = 'django.db.models.BigAutoField'
```

## 코드 설명

### 1) `INSTALLED_APPS`

여기에 등록된 앱만 Django가 프로젝트의 일부로 인식합니다.

우리가 만든 `study` 앱을 여기에 넣어야:

* 모델을 마이그레이션할 수 있고
* admin에 등록할 수 있고
* 앱 구조를 Django가 추적할 수 있습니다.

### 2) `TEMPLATES['DIRS']`

```python
'DIRS': [BASE_DIR / 'templates']
```

이 설정은 프로젝트 루트의 `templates` 폴더를 템플릿 탐색 경로로 추가합니다.
즉 나중에 `templates/base.html` 같은 공용 레이아웃을 쉽게 사용할 수 있게 됩니다.

### 3) `STATICFILES_DIRS`

```python
STATICFILES_DIRS = [BASE_DIR / 'static']
```

이 설정은 CSS, JS, 이미지 같은 정적 파일을 프로젝트 루트의 `static` 폴더에서 찾도록 합니다.
이번 교안에서는 CSS 디자인이 들어가므로 반드시 필요합니다.

### 4) `LANGUAGE_CODE`, `TIME_ZONE`

한국어 기준 실습이므로:

* 언어: `ko-kr`
* 시간대: `Asia/Seoul`

로 설정했습니다.

# 12. 먼저 데이터 구조를 생각해보기

이제 화면을 만들기 전에 먼저 생각해야 하는 것이 있습니다.

> 우리가 무엇을 저장할 것인가?

Django는 데이터 중심 프레임워크이기 때문에,
페이지보다 먼저 **모델 설계**를 생각하는 습관이 중요합니다.

이번 공부시간 기록기에서는 아래 정보를 저장해보겠습니다.

| 필드명        | 의미    | 예시                   |
| ---------- | ----- | -------------------- |
| title      | 공부 제목 | Django ORM 복습        |
| category   | 카테고리  | Python, Django, 알고리즘 |
| study_date | 공부 날짜 | 2026-03-12           |
| hours      | 시간    | 2                    |
| minutes    | 분     | 30                   |
| memo       | 메모    | 오늘은 모델과 뷰의 차이를 복습했다  |
| created_at | 생성 시각 | 자동 저장                |
| updated_at | 수정 시각 | 자동 갱신                |

## 왜 모델을 먼저 생각해야 할까?

모델이 정해지면:

* DB 구조가 정해지고
* admin 화면 구조가 정해지고
* Form 구조가 정해지고
* 나중에 Serializer 구조도 정해집니다.

즉 Django에서는 아래 사고 흐름이 중요합니다.

```text
무슨 화면을 만들까?
보다
무슨 데이터를 저장할까?
```

# 13. Study 모델 작성

이제 실제 모델을 작성합니다.

```python title="study/models.py"
from django.db import models


class Study(models.Model):
    CATEGORY_CHOICES = [
        ('python', 'Python'),
        ('django', 'Django'),
        ('algorithm', '알고리즘'),
        ('database', 'Database'),
        ('etc', '기타'),
    ]

    title = models.CharField(max_length=200, verbose_name='공부 제목')
    category = models.CharField(max_length=30, choices=CATEGORY_CHOICES, default='django', verbose_name='카테고리')
    study_date = models.DateField(verbose_name='공부 날짜')
    hours = models.PositiveIntegerField(default=0, verbose_name='시간')
    minutes = models.PositiveIntegerField(default=0, verbose_name='분')
    memo = models.TextField(blank=True, verbose_name='메모')
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['-study_date', '-created_at']
        verbose_name = '공부 기록'
        verbose_name_plural = '공부 기록들'

    def __str__(self):
        return f'{self.title} ({self.study_date})'

    @property
    def total_minutes(self):
        return self.hours * 60 + self.minutes
```

## 코드 설명

### 1) `title = models.CharField(...)`

공부 제목을 저장하는 문자열 필드입니다.
예:

* Django ORM 복습
* Python 기초 문법 정리
* SQL JOIN 학습

`max_length=200`은 최대 글자 수입니다.

### 2) `category = models.CharField(..., choices=...)`

카테고리는 자유 입력이 아니라 정해진 선택지 안에서 고르게 했습니다.
이렇게 하면:

* 오타를 줄일 수 있고
* admin과 폼에서 선택 UI가 자동으로 나오고
* 데이터가 더 깔끔해집니다.

### 3) `study_date = models.DateField(...)`

공부한 날짜를 저장합니다.
언제 공부했는지 기록하는 데 필요합니다.

### 4) `hours`, `minutes`

공부 시간을 한 필드에 넣지 않고,

* 시간
* 분

으로 분리했습니다.

이 방식은 초보자가 이해하기 쉽고, 나중에 입력 폼도 직관적으로 만들 수 있습니다.

### 5) `memo = models.TextField(blank=True)`

메모는 길게 작성할 수 있으므로 `TextField`를 사용했습니다.
`blank=True`는 비워둘 수 있다는 의미입니다.

### 6) `created_at`, `updated_at`

* `auto_now_add=True`: 처음 생성될 때 자동 저장
* `auto_now=True`: 수정될 때마다 자동 갱신

이 두 필드는 나중에 정렬이나 데이터 추적에 자주 쓰입니다.

### 7) `class Meta`

모델의 부가 설정입니다.

* `ordering`: 최신 날짜 순 정렬
* `verbose_name`: admin에서 보이는 이름

### 8) `__str__`

관리자 페이지나 Django shell에서 객체를 사람이 보기 좋은 문자열로 보여주기 위해 사용합니다.
이게 없으면 `Study object (1)`처럼 보일 수 있습니다.

### 9) `total_minutes`

공부 시간을 총 분으로 계산하는 속성입니다.
예를 들어 2시간 30분이면 150을 반환합니다.

이처럼 **모델 안에 데이터 관련 계산 로직**을 넣는 패턴도 자주 사용됩니다.

# 14. 마이그레이션: 모델을 DB에 반영하기

모델을 만들었다고 해서 DB에 테이블이 바로 생기는 것은 아닙니다.
반드시 마이그레이션을 실행해야 합니다.

## 1) migration 파일 생성

```bash title="terminal"
python manage.py makemigrations
```

## 2) 실제 DB 반영

```bash title="terminal"
python manage.py migrate
```

## 이 작업이 의미하는 것

```text
models.py에 모델 작성
→ Django가 변경사항 감지
→ migration 파일 생성
→ 실제 DB(SQLite)에 테이블 반영
```

## 초보자가 가장 많이 하는 오해

> 모델만 만들면 DB에 자동으로 들어간다

아닙니다.
Django는 안전하게 DB를 관리하기 위해 아래 단계를 분리합니다.

* `makemigrations`: 설계서 만들기
* `migrate`: 실제 DB에 적용하기

둘 다 해야 합니다.

## 실행 후 꼭 확인할 것

* 프로젝트 루트에 `db.sqlite3` 파일이 생겼는지
* `study/migrations/` 폴더에 migration 파일이 생겼는지

# 15. admin 등록하기

이제 관리자 페이지에서 `Study` 데이터를 직접 확인하고 관리할 수 있도록 등록합니다.

```python title="study/admin.py"
from django.contrib import admin
from .models import Study


@admin.register(Study)
class StudyAdmin(admin.ModelAdmin):
    list_display = (
        'id',
        'title',
        'category',
        'study_date',
        'hours',
        'minutes',
        'created_at',
    )
    list_filter = ('category', 'study_date')
    search_fields = ('title', 'memo')
    ordering = ('-study_date', '-created_at')
```

## 코드 설명

### `list_display`

admin 목록 화면에서 어떤 컬럼을 보여줄지 정합니다.

### `list_filter`

오른쪽 필터 영역에서 카테고리나 날짜 기준으로 데이터를 좁혀볼 수 있습니다.

### `search_fields`

검색창에서 제목이나 메모 내용을 검색할 수 있게 합니다.

### `ordering`

최신 공부 기록이 위로 오도록 정렬했습니다.

즉 admin도 단순 등록만 하지 않고,
**조금 더 보기 좋고 실무적으로 쓰기 편하게** 설정한 것입니다.

# 23. 공용 레이아웃 base.html 만들기

```html title="templates/base.html"
{% load static %}
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{% block title %}공부시간 기록기{% endblock %}</title>
  <link rel="preconnect" href="https://fonts.googleapis.com">

  <link href="https://fonts.googleapis.com/css2?family=Pretendard:wght@400;500;600;700;800&display=swap" rel="stylesheet">
  <link rel="stylesheet" href="{% static 'css/style.css' %}">
</head>
<body>
  <div class="app-shell">
    <header class="site-header">
      <div class="header-inner container">
        <a href="{% url 'study:list' %}" class="brand">Study Tracker</a>
        <nav class="main-nav">
          <a href="{% url 'study:list' %}">홈</a>
        </nav>
      </div>
    </header>

    <main class="site-main">
      {% block content %}{% endblock %}
    </main>

    <footer class="site-footer">
      <div class="container footer-inner">
        <p>꾸준한 기록이 결국 큰 성장을 만듭니다.</p>
        <p class="footer-sub">Django 반복훈련 1 - 공부시간 기록기</p>
      </div>
    </footer>
  </div>
</body>
</html>
```

## 코드 설명

### `{% load static %}`

정적 파일 태그를 사용하기 위해 필요합니다.
이게 있어야 CSS 파일을 Django 문법으로 불러올 수 있습니다.

### `{% block title %}`, `{% block content %}`

하위 페이지에서 내용을 갈아끼울 수 있는 영역입니다.

즉 `base.html`은 공통 틀이고,
각 페이지는 이 틀을 상속해서 자신만의 내용을 넣게 됩니다.

## 왜 base.html을 먼저 만들까?

초보자가 페이지마다 HTML 전체를 매번 쓰면 아래 문제가 생깁니다.

* header 중복
* footer 중복
* CSS 연결 중복
* 레이아웃 수정 어려움

그래서 Django 템플릿에서는 보통 아래 구조를 많이 사용합니다.

```text
base.html 공통 틀
→ 각 페이지가 extends로 상속
```

# 25. 함수형 View로 목록 페이지 만들기

이제 진짜 데이터를 가져와 HTML로 보여줍니다.

이번 편에서는 가장 기본인 **함수형 View(FBV)** 로 목록 페이지를 구현합니다.

```python title="study/views.py"
from django.shortcuts import render
from .models import Study


def study_list(request):
    studies = Study.objects.all()

    total_count = studies.count()
    total_minutes = sum(study.total_minutes for study in studies)
    total_hours = total_minutes // 60
    remain_minutes = total_minutes % 60

    context = {
        'studies': studies,
        'total_count': total_count,
        'total_hours': total_hours,
        'remain_minutes': remain_minutes,
    }
    return render(request, 'study/study_list.html', context)
```

## 코드 설명

### `studies = Study.objects.all()`

DB에 저장된 모든 공부 기록을 조회합니다.

### 요약 데이터 계산

단순 목록만 보여주는 것이 아니라,
아래 정보도 함께 계산합니다.

* 전체 기록 개수
* 전체 공부 시간을 분 단위로 합산
* 다시 시간과 분으로 나누기

즉 View는 단순히 DB 조회만 하는 곳이 아니라,
**템플릿에 전달할 화면용 데이터를 준비하는 곳**이기도 합니다.

### `context`

템플릿에 전달할 데이터를 딕셔너리로 모아둔 것입니다.

### `render(request, 'study/study_list.html', context)`

Django에서 HTML 화면을 반환하는 기본 패턴입니다.

즉 아래 흐름을 꼭 기억하세요.

```text
데이터 조회
→ 화면용 데이터 준비
→ context에 담기
→ render로 템플릿에 전달
```

# 29. 자주 하는 실수 정리

## 1) `study` 앱을 `INSTALLED_APPS`에 등록하지 않음

이 경우 마이그레이션이나 admin 등록이 정상적으로 되지 않을 수 있습니다.

## 2) 모델 작성 후 `makemigrations`, `migrate`를 안 함

코드만 바뀌고 실제 DB는 그대로라서 혼란이 생깁니다.

## 3) admin 등록을 안 함

관리자 페이지에서 데이터 확인이 되지 않습니다.

## 4) `templates` / `static` 설정을 안 넣음

Template나 CSS 연결이 꼬일 수 있습니다.

## 5) `study/urls.py`를 만들었지만 `config/urls.py`에서 include하지 않음

브라우저에서 해당 View가 열리지 않습니다.

## 6) View에서 context로 넘긴 이름과 템플릿에서 쓰는 이름이 다름

예를 들어 View는 `studies`로 넘겼는데 템플릿에서 `study_list`라고 쓰면 데이터가 안 보입니다.
