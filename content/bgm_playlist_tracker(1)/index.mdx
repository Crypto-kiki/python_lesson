# Django 기초 - BGM 플레이리스트 기록기 만들기(1) : 프로젝트 생성, 모델 작성, 함수형 목록 페이지

이번 교안에서는 **내가 좋아하는 곡들을 직접 기록하고 모아볼 수 있는 Django 웹앱**을 만들어 봅니다.

기존의 TodoList, 메모장처럼 너무 익숙한 주제 대신,
조금 더 결과물이 눈에 보이고 감성적인 프로젝트로 **BGM 플레이리스트 기록기**를 선택했습니다.

이번 프로젝트에서 사용자는 아래 정보를 저장하게 됩니다.

* 가수
* 곡 제목
* 별점(1~5)
* 한줄 코멘트
* 유튜브 링크(선택)

그리고 1편에서는 우선 아래 흐름까지 완성합니다.

```
프로젝트 생성
→ 앱 생성
→ settings 등록
→ 모델 작성
→ 마이그레이션
→ admin 등록
→ 함수형 View 목록 페이지
→ 카드형 메인 화면 완성
```

즉 이번 1편의 목표는
**“곡 데이터를 저장할 수 있는 구조를 만들고, 메인 홈페이지에 카드 목록으로 예쁘게 보여주는 것”** 입니다.

---

# 0. 이번 교안의 진짜 목표

이번 교안의 핵심은 단순히 노래 정보를 몇 개 저장하는 데 있지 않습니다.

실제로는 아래 3가지를 훈련하는 것이 더 중요합니다.

| 목표                    | 설명                                                         |
| --------------------- | ---------------------------------------------------------- |
| Django 프로젝트 시작 흐름 익히기 | 프로젝트 생성 → 앱 생성 → settings 등록 → URL 연결 흐름을 자연스럽게 익힌다        |
| 모델 중심 개발 흐름 이해        | Song 모델을 만들고, migration과 admin까지 연결해보며 Django의 기본 구조를 이해한다 |
| 템플릿 기반 목록 페이지 만들기     | 함수형 View로 데이터를 조회하고, 템플릿에서 카드형 UI로 출력하는 흐름을 익힌다            |

즉 이번 교안에서 꼭 머릿속에 남겨야 할 흐름은 아래입니다.

```
무엇을 저장할지 먼저 정한다
→ 모델을 만든다
→ DB에 반영한다
→ 관리자 페이지에서 데이터를 넣어본다
→ View에서 조회한다
→ Template에서 화면으로 보여준다
```

이 흐름이 익숙해지면, 이후 2편에서 만들 **상세 / 생성 / 수정 / 삭제**,
3편에서 만들 **검색 / 필터 / 추천 기능**도 훨씬 덜 어렵게 느껴집니다.

## 왜 이 순서인가?

이 순서는 그냥 편한 순서가 아니라,
**Django가 실제로 동작하는 흐름에 맞춘 순서**입니다.

예를 들면:

* 모델이 없으면 migration을 할 수 없습니다.
* migration이 안 되어 있으면 admin에서 데이터를 넣을 수 없습니다.
* View가 있어야 URL을 연결할 수 있습니다.
* View와 URL이 연결되어야 템플릿을 브라우저에서 볼 수 있습니다.

즉 Django는 아래처럼 **앞 단계가 준비되어 있어야 다음 단계가 가능한 구조**입니다.

```
모델 작성
→ DB 반영
→ 관리자에서 데이터 입력
→ View 조회
→ Template 출력
```

이 흐름을 이번 1편에서 확실히 잡아두는 것이 가장 중요합니다.

# 5. settings.py에 앱 등록하기

Django는 앱을 만들었다고 해서 자동으로 사용하는 것이 아닙니다.
반드시 `INSTALLED_APPS`에 등록해야 합니다.

```python title="config/settings.py"
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
    'music',
]
```

## 왜 등록해야 할까?

앱을 등록해야 Django가 아래 작업을 할 수 있습니다.

* 모델 인식
* migration 대상 포함
* admin 연결
* 템플릿 / static 탐색 구조 반영

즉 `music` 앱을 만들었으면 거의 항상 `INSTALLED_APPS` 등록이 뒤따라야 합니다.

# 6. Song 모델 작성하기

이제 이번 프로젝트에서 저장할 핵심 데이터 구조를 정의하겠습니다.

이번 프로젝트의 입력값은 아래와 같습니다.

* 가수
* 곡 제목
* 별점(1~5)
* 한줄 코멘트
* 유튜브 링크(선택)

여기에 Django가 자주 쓰는 `생성일`, `수정일`도 같이 넣겠습니다.

```python title="music/models.py"
from django.core.validators import MaxValueValidator, MinValueValidator
from django.db import models


class Song(models.Model):
    artist = models.CharField(max_length=100, verbose_name="가수")
    title = models.CharField(max_length=100, verbose_name="곡 제목")
    rating = models.PositiveSmallIntegerField(
        verbose_name="별점",
        validators=[MinValueValidator(1), MaxValueValidator(5)],
    )
    comment = models.TextField(verbose_name="한줄 코멘트")
    youtube_url = models.URLField(
        blank=True,
        verbose_name="유튜브 링크",
        help_text="선택 입력입니다. 2편, 3편에서 활용합니다.",
    )
    created_at = models.DateTimeField(auto_now_add=True, verbose_name="생성일")
    updated_at = models.DateTimeField(auto_now=True, verbose_name="수정일")

    class Meta:
        ordering = ["-created_at"]
        verbose_name = "곡"
        verbose_name_plural = "곡 목록"

    def __str__(self):
        return f"{self.artist} - {self.title}"
```

---

## 필드 하나씩 해석하기

### `artist`

```python title="music/models.py"
artist = models.CharField(max_length=100, verbose_name="가수")
```

짧은 문자열을 저장할 때 사용하는 필드입니다.
가수 이름은 보통 길지 않으므로 `CharField`가 적절합니다.

---

### `title`

```python title="music/models.py"
title = models.CharField(max_length=100, verbose_name="곡 제목")
```

곡 제목도 짧은 텍스트이므로 `CharField`를 사용합니다.

---

### `rating`

```python title="music/models.py"
rating = models.PositiveSmallIntegerField(
    verbose_name="별점",
    validators=[MinValueValidator(1), MaxValueValidator(5)],
)
```

별점은 1~5 사이의 숫자만 허용하고 싶기 때문에
정수 필드를 사용하고, validator로 범위를 제한합니다.

이렇게 해두면 잘못된 숫자가 들어가는 것을 줄일 수 있습니다.

---

### `comment`

```python title="music/models.py"
comment = models.TextField(verbose_name="한줄 코멘트")
```

한줄 코멘트라고 이름은 붙였지만,
실제로는 짧은 감상평이나 추천 이유가 들어갈 수 있으므로 `TextField`를 사용합니다.

`CharField`와 달리 길이 제한을 엄격하게 잡지 않아도 되므로 조금 더 유연합니다.

---

### `youtube_url`

```python title="music/models.py"
youtube_url = models.URLField(
    blank=True,
    verbose_name="유튜브 링크",
    help_text="선택 입력입니다. 2편, 3편에서 활용합니다.",
)
```

유튜브 링크는 나중에 상세페이지 재생, 썸네일 추출에 활용할 예정입니다.

여기서 중요한 점은 `blank=True` 입니다.

즉, 지금은 링크를 입력하지 않아도 저장이 가능하게 해두는 것입니다.

이렇게 해두면:

* 입력 부담이 줄어들고
* 링크 없는 데이터도 저장 가능하며
* 나중에 조건부 렌더링을 설명하기 좋습니다.

---

### `created_at`, `updated_at`

```python title="music/models.py"
created_at = models.DateTimeField(auto_now_add=True, verbose_name="생성일")
updated_at = models.DateTimeField(auto_now=True, verbose_name="수정일")
```

이 두 필드는 Django 프로젝트에서 매우 자주 쓰는 기본 필드입니다.

* `auto_now_add=True` → 처음 생성될 때만 자동 입력
* `auto_now=True` → 수정될 때마다 자동 갱신

이번 1편에서는 화면에 꼭 보여주지 않아도 되지만,
관리자 페이지나 정렬 기준에서 매우 유용합니다.

---

### `Meta`

```python title="music/models.py"
class Meta:
    ordering = ["-created_at"]
    verbose_name = "곡"
    verbose_name_plural = "곡 목록"
```

`ordering = ["-created_at"]`는 **최신 등록 곡이 먼저 보이도록** 정렬하는 설정입니다.

즉, 별도의 정렬 코드를 매번 쓰지 않아도 기본적으로 최신 데이터가 위로 옵니다.

---

### `__str__`

```python title="music/models.py"
def __str__(self):
    return f"{self.artist} - {self.title}"
```

이 코드를 넣어두면 admin 페이지, shell 등에서 객체를 볼 때
`Song object(1)`처럼 밋밋하게 보이지 않고, 사람이 읽기 좋은 형태로 표시됩니다.

# 7. 마이그레이션하기

모델을 만들었다고 해서 바로 DB에 테이블이 생기는 것은 아닙니다.
반드시 migration 과정을 거쳐야 합니다.

```bash title="terminal"
python manage.py makemigrations
python manage.py migrate
```

## 두 명령어 차이

| 명령어              | 역할                          |
| ---------------- | --------------------------- |
| `makemigrations` | 모델 변경 내용을 migration 파일로 생성  |
| `migrate`        | 생성된 migration 내용을 실제 DB에 반영 |

즉 흐름은 늘 아래처럼 기억하면 됩니다.

```
models.py 수정
→ makemigrations
→ migrate
```

```python title="music/admin.py"
from .models import Song


@admin.register(Song)
class SongAdmin(admin.ModelAdmin):

    list_filter = ("rating", "artist")
    search_fields = ("artist", "title", "comment")
```

```python title="music/admin.py"
list_filter = ("rating", "artist")
```

검색창에서 어떤 필드를 대상으로 검색할지 지정합니다.

```python title="music/admin.py"
search_fields = ("artist", "title", "comment")
```

# 11. 함수형 View로 목록 페이지 만들기

이제 브라우저 메인 화면에서 곡 목록을 보여주겠습니다.

우선 `music/views.py`에 함수형 View를 작성합니다.

```python title="music/views.py"
from django.shortcuts import render
from .models import Song


def song_list(request):
    songs = Song.objects.all()
    return render(request, "music/song_list.html", {"songs": songs})
```

## 코드 설명

### `Song.objects.all()`

DB에 저장된 모든 곡 데이터를 가져옵니다.

```python title="music/views.py"
songs = Song.objects.all()
```

이번 모델의 `Meta.ordering = ["-created_at"]` 덕분에
최신 등록 순으로 자동 정렬됩니다.

---

### `render()`

`music/song_list.html` 템플릿에 데이터를 전달해서 브라우저 화면으로 보여줍니다.

```python title="music/views.py"
return render(request, "music/song_list.html", {"songs": songs})
```

여기서 `songs`라는 이름으로 템플릿에 전달되므로,
HTML에서는 `{% for song in songs %}` 형태로 반복문을 사용할 수 있습니다.

# 15. settings.py에 templates / static 경로 설정하기

프로젝트 루트의 `templates`, `static` 폴더를 쓰기 위해 설정을 조금 추가하겠습니다.

```python title="config/settings.py"
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent.parent

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [BASE_DIR / "templates"],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.request',

STATIC_URL = "static/"
STATICFILES_DIRS = [
    BASE_DIR / "static",
```

```python title="config/settings.py"
'DIRS': [BASE_DIR / "templates"],
```

이 설정이 있어야 CSS 같은 정적 파일을 프로젝트 루트 `static/` 폴더에서 찾을 수 있습니다.

## 이 템플릿에서 중요한 포인트

### `{% extends "music/base.html" %}`

공통 레이아웃을 상속받는다는 의미입니다.


```html title="templates/music/song_list.html"
{% if songs %}
...
{% else %}
...
{% endif %}
```

이런 조건 분기는 Django 템플릿에서 매우 자주 사용합니다.

---

### `{% for song in songs %}`

전달받은 곡 목록을 반복 출력합니다.

```html title="templates/music/song_list.html"
{% for song in songs %}
```

이 부분이 바로:

```python
{"songs": songs}
```

와 연결되는 지점입니다.

---

```html title="templates/music/song_list.html"
{% for _ in "12345" %}
  {% if forloop.counter <= song.rating %}

  {% else %}
    <span class="star">☆</span>
  {% endif %}
{% endfor %}
```

별점 1~5를 숫자로만 보여주면 밋밋할 수 있기 때문에,
별 모양으로 시각화했습니다.

# 21. 자주 만나는 오류 정리

이번 단계에서 초보자가 가장 자주 막히는 포인트를 정리해보겠습니다.

## 1) `No module named 'music.urls'`

보통 `music/urls.py` 파일을 아직 만들지 않았거나,
파일 이름이 다를 때 발생합니다.

반드시 아래 파일이 있어야 합니다.

```python title="music/urls.py"
from django.urls import path
from . import views

app_name = "music"

urlpatterns = [
    path("", views.song_list, name="song_list"),
]
```

---

## 2) `TemplateDoesNotExist`

보통 템플릿 경로가 잘못되었을 때 발생합니다.

이번 교안에서는 아래 경로를 사용합니다.

```text title="template path"
templates/music/base.html
templates/music/song_list.html
```

그리고 `settings.py` 에도 아래 설정이 있어야 합니다.

```python title="config/settings.py"
'DIRS': [BASE_DIR / "templates"],
```

---

## 3) CSS가 적용되지 않음

아래 항목을 체크해보세요.

* `static/music/css/style.css` 경로가 맞는지
* `base.html` 상단에 `{% load static %}` 가 있는지
* `<link rel="stylesheet" href="{% static 'music/css/style.css' %}" />` 문법이 맞는지
* `STATICFILES_DIRS` 설정이 들어갔는지

---

## 4) admin에 모델이 안 보임

`music/admin.py` 등록을 안 했을 가능성이 큽니다.

```python title="music/admin.py"
@admin.register(Song)
class SongAdmin(admin.ModelAdmin):
    ...
```

또는 `INSTALLED_APPS`에 `music` 등록이 안 되어 있을 수도 있습니다.

---

## 5) `OperationalError: no such table`

이건 거의 항상 migration 문제입니다.

모델을 만들고 아래 명령어를 하지 않았을 가능성이 큽니다.

```bash title="terminal"
python manage.py makemigrations
python manage.py migrate
```

# 22. 이번 편 핵심 개념 요약

이번 1편에서 가장 중요한 개념은 아래 5가지입니다.

| 개념        | 핵심                                        |
| --------- | ----------------------------------------- |
| 앱 등록      | 만든 앱은 `INSTALLED_APPS`에 등록해야 Django가 인식한다 |
| 모델        | 저장할 데이터 구조를 `models.py`에서 정의한다            |
| migration | 모델을 실제 DB 테이블로 반영하는 과정이다                  |
| admin     | 코드를 일일이 안 짜고도 데이터를 쉽게 넣어볼 수 있다            |
| 함수형 View  | DB 데이터를 조회해서 템플릿으로 넘기는 가장 직관적인 방식이다       |

즉 오늘은 아래 구조를 실제로 손으로 만든 셈입니다.

```
Model
→ Migration
→ Admin
→ View
→ Template
→ CSS
```

이 순서를 반복해서 익히는 것이 Django 기초에서 정말 중요합니다.

# 23. 체크리스트

아래 항목이 모두 되면 이번 1편은 성공입니다.

* ✅ Django 프로젝트와 `music` 앱을 생성했다
* ✅ `settings.py`에 `music` 앱을 등록했다
* ✅ `Song` 모델을 작성했다
* ✅ `makemigrations`, `migrate`를 실행했다
* ✅ `createsuperuser`로 관리자 계정을 만들었다
* ✅ admin에 `Song` 모델을 등록했다
* ✅ 관리자 페이지에서 곡 데이터를 직접 추가해봤다
* ✅ 함수형 View로 곡 목록 페이지를 만들었다
* ✅ `/` 접속 시 곡 카드 목록이 보인다
* ✅ 카드 UI가 3열 그리드 형태로 잘 보인다

# 24. 결론

이번 1편에서는 BGM 플레이리스트 프로젝트의 가장 중요한 뼈대를 완성했습니다.

아직 사용자가 브라우저에서 직접 곡을 등록하는 기능은 없지만,
그 전에 더 중요한 것들을 이미 끝냈습니다.

* 어떤 데이터를 저장할지 정했다
* Django 모델로 구조를 만들었다
* DB에 반영했다
* 관리자 페이지에서 데이터를 넣어봤다
* 브라우저 메인 화면에 목록을 출력했다

즉 지금 상태는 이미 아래 흐름이 완성된 것입니다.

```
데이터 저장 구조 완성
→ 데이터 직접 입력 가능
→ 메인 목록 화면 출력 가능
```

다음 2편에서는 여기서 한 단계 더 나아가서
**곡 등록 / 상세 / 수정 / 삭제**를 붙이며 CRUD를 완성할 예정입니다.

그때부터는 지금 만든 목록 페이지가 진짜 웹앱처럼 살아나기 시작합니다.
