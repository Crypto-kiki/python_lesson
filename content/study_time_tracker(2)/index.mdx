# 0. 이번 교안의 진짜 목표

이번 교안의 핵심 목표는 단순히 버튼 몇 개를 추가하는 것이 아닙니다.

실제로는 아래 3가지를 훈련하는 것이 중요합니다.

| 목표                  | 설명                                             |
| ------------------- | ---------------------------------------------- |
| ModelForm 흐름 익히기    | Django에서 입력값을 검증하고 저장하는 표준 흐름 이해               |
| 함수형 View 기준 CRUD 완성 | 생성, 상세, 수정, 삭제가 각각 어떻게 구현되는지 체득                |
| 브라우저 중심 데이터 흐름 이해   | HTML 폼 입력 → POST 요청 → 검증 → 저장 → redirect 흐름 이해 |

즉 이번 편에서는 아래 흐름을 머릿속에 남기는 것이 핵심입니다.

```
사용자가 브라우저에서 값 입력
→ Django View가 POST 요청 받음
→ ModelForm이 검증
→ 모델에 저장
→ redirect로 화면 이동
→ 목록 / 상세 페이지에서 결과 확인
```

이 흐름이 이해되어야 다음 교안에서 클래스형 View, 제너릭 View, DRF까지 자연스럽게 연결됩니다.

# 4. Django Form / ModelForm은 왜 필요한가?

이제 본격적으로 폼을 만들기 전에,
왜 Django에서 Form 또는 ModelForm을 쓰는지 먼저 이해하고 갑니다.

초보자는 자주 이렇게 생각합니다.

> HTML form 만들고 request.POST에서 값 꺼내서 저장하면 되는 거 아닌가요?

물론 기술적으로는 가능합니다.
하지만 Django는 Form을 쓰는 방식이 훨씬 체계적이고 안전합니다.

## Form / ModelForm을 쓰는 이유

* 입력값 검증이 쉬움
* 에러 메시지 처리 쉬움
* 필드 관리가 쉬움
* 템플릿 렌더링이 쉬움
* 코드 구조가 Django 표준 흐름에 맞음

특히 `ModelForm`은 모델과 직접 연결되어 있어서 아주 강력합니다.

즉 아래 흐름이 가능해집니다.

```
Study 모델 정의
→ ModelForm 작성
→ 폼에서 입력값 검증
→ form.save()로 모델 저장
```

즉 ModelForm은 **모델 기반 폼 자동화 도구**라고 이해하면 좋습니다.

# 5. ModelForm 만들기

이제 실제로 `Study` 모델과 연결되는 `StudyForm`을 만듭니다.

```python title="study/forms.py"
from django import forms
from .models import Study


class StudyForm(forms.ModelForm):
    class Meta:
        model = Study
        fields = ['title', 'category', 'study_date', 'hours', 'minutes', 'memo']
        widgets = {
            'title': forms.TextInput(attrs={
                'placeholder': '예: Django ORM 복습',
            }),
            'category': forms.Select(),
            'study_date': forms.DateInput(attrs={
                'type': 'date',
            }),
            'hours': forms.NumberInput(attrs={
                'min': 0,
                'placeholder': '예: 2',
            }),
            'minutes': forms.NumberInput(attrs={
                'min': 0,
                'max': 59,
                'placeholder': '예: 30',
            }),
            'memo': forms.Textarea(attrs={
                'rows': 6,
                'placeholder': '오늘 공부한 내용, 어려웠던 점, 다시 복습할 포인트를 적어보세요.',
            }),
        }
        labels = {
            'title': '공부 제목',
            'category': '카테고리',
            'study_date': '공부 날짜',
            'hours': '시간',
            'minutes': '분',
            'memo': '메모',
        }

    def clean_minutes(self):
        minutes = self.cleaned_data['minutes']
        if minutes >= 60:
            raise forms.ValidationError('분은 0 이상 59 이하로 입력해주세요.')
        return minutes
```

## 코드 설명

### 1) `class StudyForm(forms.ModelForm)`

`StudyForm`은 `Study` 모델 기반의 ModelForm입니다.
즉 어떤 필드를 입력받을지, 어떤 모델에 저장할지를 Django에게 알려주는 구조입니다.

### 2) `fields`

폼에서 입력받을 필드를 지정합니다.
즉 사용자가 입력할 수 있는 항목을 여기서 고릅니다.

### 3) `widgets`

각 입력 필드의 HTML 형태를 조금 더 예쁘고 직관적으로 제어합니다.
예를 들어:

* `study_date`는 달력 입력처럼 보이게
* `memo`는 큰 textarea로 보이게
* `hours`, `minutes`는 숫자 입력처럼 보이게

만듭니다.

### 4) `labels`

기본 영어 필드명을 그대로 노출하지 않고,
사용자에게 보이는 이름을 한국어로 자연스럽게 지정합니다.

### 5) `clean_minutes()`

분은 60 이상이면 이상하므로,
직접 유효성 검사를 추가했습니다.

즉 ModelForm은 단순히 입력창 생성뿐 아니라,
**검증 로직까지 포함하는 구조**입니다.

# 7. 생성(Create) View 작성

이제 `study_create` View를 추가합니다.

```python title="study/views.py"
from django.shortcuts import redirect, render
from .forms import StudyForm
from .models import Study


def study_list(request):
    studies = Study.objects.all()

    total_count = studies.count()
    total_minutes = sum(study.total_minutes for study in studies)
    total_hours = total_minutes // 60
    remain_minutes = total_minutes % 60

    context = {
        'studies': studies,
        'total_count': total_count,
        'total_hours': total_hours,
        'remain_minutes': remain_minutes,
    }
    return render(request, 'study/study_list.html', context)


def study_create(request):
    form = StudyForm(request.POST or None)

    if request.method == 'POST' and form.is_valid():
        form.save()
        return redirect('study:list')

    return render(request, 'study/study_form.html', {
        'form': form,
        'page_title': '공부 기록 작성',
    })
```

## 코드 설명

### `form = StudyForm(request.POST or None)`

이 문장은 초보자가 처음 보면 낯설 수 있습니다.

* GET 요청이면 `request.POST`는 비어 있으므로 빈 폼 생성
* POST 요청이면 사용자가 제출한 값으로 폼 생성

즉 하나의 코드로 GET / POST 모두 처리하는 패턴입니다.

### `if request.method == 'POST' and form.is_valid():`

이 부분이 생성 기능의 핵심입니다.

```
사용자가 폼 제출함
→ POST 요청 들어옴
→ 폼 검증
→ 유효하면 저장
```

### `form.save()`

ModelForm이기 때문에 이 한 줄로 `Study` 모델에 저장할 수 있습니다.

### `redirect('study:list')`

저장 후 다시 목록 페이지로 이동시킵니다.

## `render()`와 `redirect()` 차이

이 부분은 꼭 이해해야 합니다.

### `render()`

템플릿을 바로 보여줄 때 사용합니다.

### `redirect()`

다른 URL로 다시 이동시킬 때 사용합니다.

즉 생성 완료 후에는 다시 목록 페이지를 보여주고 싶으므로 `redirect()`를 사용합니다.

## 코드 설명

### `{% csrf_token %}`

Django의 POST 폼에서는 CSRF 보안 토큰이 필요합니다.
이게 없으면 폼 제출 시 보안 오류가 날 수 있습니다.

즉 Django에서 POST 폼을 만들 때는 아래를 거의 항상 같이 기억해야 합니다.

```html
<form method="post">
  {% csrf_token %}
</form>
```

### 필드를 하나씩 직접 출력한 이유

Django Form은 `{{ form.as_p }}`처럼 한 번에 출력할 수도 있습니다.
하지만 이번 교안에서는:

* 디자인을 더 세밀하게 제어하고
* 수강생이 구조를 더 잘 이해하고
* 에러 메시지를 보기 좋게 배치하기 위해

필드를 하나씩 직접 렌더링했습니다.

### 왜 생성/수정 템플릿을 하나로 두는가?

생성과 수정은 화면 구조가 거의 같습니다.
둘 다 입력 폼을 보여준다는 점에서 동일하기 때문입니다.

차이는:

* 생성: 빈 값으로 시작
* 수정: 기존 값이 채워진 상태로 시작

이 차이만 View에서 처리하면 템플릿은 공용으로 재사용할 수 있습니다.

# 12. 상세(Detail) View 작성

이제 특정 공부기록 하나를 가져와서 보여주는 View를 작성합니다.

```python title="study/views.py"
from django.shortcuts import get_object_or_404, redirect, render
from .forms import StudyForm
from .models import Study


def study_list(request):
    studies = Study.objects.all()

    total_count = studies.count()
    total_minutes = sum(study.total_minutes for study in studies)
    total_hours = total_minutes // 60
    remain_minutes = total_minutes % 60

    context = {
        'studies': studies,
        'total_count': total_count,
        'total_hours': total_hours,
        'remain_minutes': remain_minutes,
    }
    return render(request, 'study/study_list.html', context)


def study_create(request):
    form = StudyForm(request.POST or None)

    if request.method == 'POST' and form.is_valid():
        form.save()
        return redirect('study:list')

    return render(request, 'study/study_form.html', {
        'form': form,
        'page_title': '공부 기록 작성',
    })


def study_detail(request, pk):
    study = get_object_or_404(Study, pk=pk)
    return render(request, 'study/study_detail.html', {
        'study': study,
    })
```

## 코드 설명

### `get_object_or_404(Study, pk=pk)`

이 코드는 아주 중요합니다.

의미는 아래와 같습니다.

* 해당 pk를 가진 `Study` 객체를 찾기
* 있으면 가져오기
* 없으면 404 에러 페이지 보여주기

즉 데이터가 없을 때 서버가 이상하게 터지는 대신,
사용자에게 “없는 페이지”라는 의미의 404를 보여주는 안전한 방식입니다.

이 패턴은 Django에서 정말 자주 씁니다.

# 16. 수정(Update) View 작성

이제 수정 기능을 작성합니다.

```python title="study/views.py"
from django.shortcuts import get_object_or_404, redirect, render
from .forms import StudyForm
from .models import Study


def study_list(request):
    studies = Study.objects.all()

    total_count = studies.count()
    total_minutes = sum(study.total_minutes for study in studies)
    total_hours = total_minutes // 60
    remain_minutes = total_minutes % 60

    context = {
        'studies': studies,
        'total_count': total_count,
        'total_hours': total_hours,
        'remain_minutes': remain_minutes,
    }
    return render(request, 'study/study_list.html', context)


def study_create(request):
    form = StudyForm(request.POST or None)

    if request.method == 'POST' and form.is_valid():
        form.save()
        return redirect('study:list')

    return render(request, 'study/study_form.html', {
        'form': form,
        'page_title': '공부 기록 작성',
    })


def study_detail(request, pk):
    study = get_object_or_404(Study, pk=pk)
    return render(request, 'study/study_detail.html', {
        'study': study,
    })


def study_update(request, pk):
    study = get_object_or_404(Study, pk=pk)
    form = StudyForm(request.POST or None, instance=study)

    if request.method == 'POST' and form.is_valid():
        form.save()
        return redirect('study:detail', pk=study.pk)

    return render(request, 'study/study_form.html', {
        'form': form,
        'page_title': '공부 기록 수정',
        'study': study,
    })
```

## 코드 설명

### 왜 `instance=study`가 필요한가?

이 부분이 수정 기능의 핵심입니다.

```python
form = StudyForm(request.POST or None, instance=study)
```

이 의미는:

* 새 객체를 만드는 것이 아니라
* 기존 `study` 객체를 기준으로 폼을 만든다

는 뜻입니다.

즉 수정 기능에서 `instance=...`가 없으면,
기존 데이터를 바꾸는 것이 아니라 새 데이터처럼 저장될 수도 있습니다.

### 수정 후 어디로 갈까?

수정이 끝나면 다시 상세 페이지로 보내는 흐름이 자연스럽습니다.
왜냐하면 방금 수정한 결과를 바로 확인할 수 있기 때문입니다.

# 19. 삭제(Delete) View 작성

이제 삭제 기능을 구현합니다.

```python title="study/views.py"
from django.shortcuts import get_object_or_404, redirect, render
from .forms import StudyForm
from .models import Study


def study_list(request):
    studies = Study.objects.all()

    total_count = studies.count()
    total_minutes = sum(study.total_minutes for study in studies)
    total_hours = total_minutes // 60
    remain_minutes = total_minutes % 60

    context = {
        'studies': studies,
        'total_count': total_count,
        'total_hours': total_hours,
        'remain_minutes': remain_minutes,
    }
    return render(request, 'study/study_list.html', context)


def study_create(request):
    form = StudyForm(request.POST or None)

    if request.method == 'POST' and form.is_valid():
        form.save()
        return redirect('study:list')

    return render(request, 'study/study_form.html', {
        'form': form,
        'page_title': '공부 기록 작성',
    })


def study_detail(request, pk):
    study = get_object_or_404(Study, pk=pk)
    return render(request, 'study/study_detail.html', {
        'study': study,
    })


def study_update(request, pk):
    study = get_object_or_404(Study, pk=pk)
    form = StudyForm(request.POST or None, instance=study)

    if request.method == 'POST' and form.is_valid():
        form.save()
        return redirect('study:detail', pk=study.pk)

    return render(request, 'study/study_form.html', {
        'form': form,
        'page_title': '공부 기록 수정',
        'study': study,
    })


def study_delete(request, pk):
    study = get_object_or_404(Study, pk=pk)

    if request.method == 'POST':
        study.delete()
        return redirect('study:list')

    return render(request, 'study/study_confirm_delete.html', {
        'study': study,
    })
```

## 코드 설명

### 왜 삭제를 GET으로 바로 하지 않을까?

삭제는 중요한 데이터 변경 작업입니다.
그래서 보통 아래 흐름을 씁니다.

* GET 요청: 삭제 확인 페이지 보여주기
* POST 요청: 실제 삭제 수행

이렇게 해야 사용자가 실수로 클릭했을 때 바로 삭제되지 않습니다.

즉 이 패턴은 **안전한 삭제 흐름**입니다.

# 23. 이제 페이지네이션을 왜 추가해야 할까?

지금은 데이터가 몇 개 없어서 한 페이지에 모두 보여도 괜찮습니다.
하지만 실제 서비스에서는 기록이 점점 많아질 수 있습니다.

예를 들어 공부 기록이:

* 10개
* 50개
* 100개
* 300개

가 된다면 한 페이지에 모두 출력하는 것은 불편합니다.

그래서 목록 화면을 여러 페이지로 나누어 보여주는 **페이지네이션**이 필요합니다.

즉 페이지네이션은 보기 좋음의 문제가 아니라,
**데이터가 많아졌을 때 화면을 관리하는 기본 기능**입니다.

# 24. 페이지네이션 적용한 목록 View로 수정하기

이제 목록 View를 수정해서 한 페이지에 5개씩 보이게 해보겠습니다.

```python title="study/views.py"
from django.core.paginator import Paginator
from django.shortcuts import get_object_or_404, redirect, render
from .forms import StudyForm
from .models import Study


def study_list(request):
    studies = Study.objects.all()
    paginator = Paginator(studies, 5)
    page_number = request.GET.get('page')
    page_obj = paginator.get_page(page_number)

        'total_count': total_count,
        'total_hours': total_hours,
        'remain_minutes': remain_minutes,
```

# 28. 함수형 View 기준 CRUD 정리

이번 편에서는 아직 클래스형 View나 제너릭 View로 넘어가지 않았습니다.
먼저 **함수형 View 기준으로 CRUD 전체 구조를 이해하는 것**이 핵심이기 때문입니다.

## 이번 편에서 만든 함수형 View

* `study_list`
* `study_create`
* `study_detail`
* `study_update`
* `study_delete`

각각의 역할은 아래와 같습니다.

| View           | 역할          |
| -------------- | ----------- |
| `study_list`   | 목록 보여주기     |
| `study_create` | 새 기록 생성     |
| `study_detail` | 한 기록 자세히 보기 |
| `study_update` | 기존 기록 수정    |
| `study_delete` | 삭제 확인 후 삭제  |

즉 Django CRUD의 가장 기본 형태를 함수형으로 먼저 체득한 것입니다.

# 29. 자주 하는 실수 정리

## 1) `forms.py`는 만들었는데 `views.py`에서 import를 안 함

`StudyForm`을 import하지 않으면 생성/수정 View에서 오류가 납니다.

## 2) 폼 템플릿에서 `{% csrf_token %}`를 빼먹음

POST 제출 시 CSRF 오류가 날 수 있습니다.

## 3) 수정 View에서 `instance=study`를 안 넣음

기존 데이터 수정이 아니라 새 데이터처럼 저장될 수 있습니다.

## 4) 삭제를 GET으로 바로 처리함

권장되지 않는 방식입니다. 삭제는 보통 확인 페이지를 거쳐 POST로 처리합니다.

## 5) 상세 URL에서 `pk`를 안 받음

특정 객체를 조회할 수 없습니다.

## 6) 목록 템플릿을 `studies` 기준으로 쓰다가 View는 `page_obj`로 바꿈

페이지네이션 적용 시 템플릿 변수명을 같이 맞춰줘야 합니다.

## 7) 페이지네이션 테스트를 안 함

데이터가 3개밖에 없으면 pagination UI가 안 보일 수 있습니다.
최소 6개 이상 넣고 테스트해야 합니다.
