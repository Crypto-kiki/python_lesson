# Django 기초 - 공부시간 기록기 만들기(3) : 클래스형 View, 제너릭 View, DRF API 확장

이번 교안은 1편, 2편에서 완성한 **브라우저 기반 공부시간 기록기**를 바탕으로,
이제 Django 구조를 조금 더 Django답게 정리하고, 같은 데이터를 **API로도 다룰 수 있게 확장**하는 단계입니다.

즉 1편, 2편이 아래 흐름이었다면,

```
1편
프로젝트 생성
→ 앱 생성
→ settings 등록
→ 모델 작성
→ 마이그레이션
→ admin 확인
→ URL 연결
→ Template / CSS
→ 함수형 View 목록 페이지

2편
ModelForm
→ Create
→ Detail
→ Update
→ Delete
→ Pagination
→ 브라우저 기반 CRUD 완성
```

이번 3편은 아래 흐름을 완성하는 것이 목표입니다.

```
함수형 View 복습
→ 클래스형 View
→ 제너릭 View
→ DRF 설치
→ Serializer
→ ViewSet
→ Router
→ API 테스트
```

즉 이번 교안이 끝나면 수강생은 아래를 모두 경험하게 됩니다.

* 함수형 View로 만든 CRUD 구조 복습
* 클래스형 View가 왜 필요한지 이해
* 제너릭 View가 왜 Django다운 방식인지 이해
* 같은 `Study` 데이터를 JSON API로도 다룰 수 있게 만들기
* 브라우저 화면과 API가 같은 DB를 공유한다는 점 확인

# 0. 이번 교안의 진짜 목표

이번 교안의 핵심 목표는 단순히 API 엔드포인트를 하나 더 만드는 것이 아닙니다.

실제로는 아래 3가지를 훈련하는 것이 중요합니다.

| 목표                  | 설명                                    |
| ------------------- | ------------------------------------- |
| View 구조 확장 이해       | 함수형 View → 클래스형 View → 제너릭 View 흐름 이해 |
| Django다운 CRUD 구조 이해 | 반복되는 CRUD 패턴을 Django가 어떻게 정리해주는지 이해   |
| API 확장 이해           | 같은 모델 데이터를 HTML이 아니라 JSON으로도 다루는 법 이해 |

즉 이번 교안에서 꼭 머릿속에 남겨야 하는 흐름은 아래입니다.

```
브라우저 기반 CRUD는 이미 완성됨
→ 구조를 더 정리해보고 싶다
→ 클래스형 / 제너릭 View를 배운다
→ 같은 데이터를 API로도 열어주고 싶다
→ DRF Serializer / ViewSet / Router를 사용한다
```

이 흐름이 이해되면, 이후 React / Next.js 같은 프론트엔드와 Django 백엔드를 나누는 구조도 훨씬 자연스럽게 받아들일 수 있습니다.

# 4. 함수형 View로도 충분한데 왜 클래스형 View를 배우는가?

초보자는 아주 자연스럽게 이런 생각을 하게 됩니다.

> 함수형 View로도 지금 다 되는데, 왜 클래스형 View를 또 배우나요?

정말 좋은 질문입니다.

함수형 View는 처음 배우기에 아주 좋습니다.
왜냐하면 요청이 들어왔을 때 어떤 일이 일어나는지 비교적 직관적으로 보이기 때문입니다.

예를 들어 함수형 View는 아래처럼 읽히기 쉽습니다.

```
요청 받기
→ DB 조회
→ 템플릿 렌더링
```

하지만 기능이 커질수록 아래 같은 문제가 생길 수 있습니다.

* 비슷한 코드가 반복된다.
* GET / POST 처리 구조를 재사용하기 어렵다.
* Django가 제공하는 고급 기능과 연결이 다소 불편하다.

반면 클래스형 View는:

* 공통 로직을 상속으로 재사용하기 쉽고
* HTTP 메서드별로 코드를 나누기 좋고
* Generic View와 자연스럽게 이어집니다.

즉 Django 학습 흐름은 보통 아래처럼 가는 것이 좋습니다.

```
함수형 View로 원리 이해
→ 클래스형 View로 구조화 이해
→ Generic View로 Django 패턴 익히기
```

# 5. 클래스형 View를 가장 단순한 예제로 먼저 보기

처음부터 CRUD 전체를 클래스형으로 바꾸면 오히려 복잡할 수 있습니다.
그래서 가장 단순한 목록 페이지 하나만 먼저 클래스형으로 바꿔보겠습니다.

이렇게 하면 함수형과 클래스형의 차이를 더 선명하게 볼 수 있습니다.

```python title="study/views_cbv.py"
from django.core.paginator import Paginator
from django.shortcuts import render
from django.views import View
from .models import Study


class StudyListView(View):
    def get(self, request):
        studies = Study.objects.all()
        paginator = Paginator(studies, 5)
        page_number = request.GET.get('page')
        page_obj = paginator.get_page(page_number)

        total_count = studies.count()
        total_minutes = sum(study.total_minutes for study in studies)
        total_hours = total_minutes // 60
        remain_minutes = total_minutes % 60

        context = {
            'page_obj': page_obj,
            'total_count': total_count,
            'total_hours': total_hours,
            'remain_minutes': remain_minutes,
        }
        return render(request, 'study/study_list.html', context)
```

## 코드 설명

### `class StudyListView(View)`

Django의 기본 `View` 클래스를 상속받아 새 View 클래스를 만듭니다.

### `def get(self, request)`

GET 요청이 들어왔을 때 실행되는 메서드입니다.

즉 함수형 View의:

```
study_list(request)
```

를 클래스 문법으로 표현한 것입니다.

# 6. 클래스형 View URL 연결 바꿔보기

이제 목록 URL만 클래스형 View로 연결해봅니다.

```python title="study/urls.py"
from django.urls import path
from . import views
from .views_cbv import StudyListView

app_name = 'study'

urlpatterns = [
    path('', StudyListView.as_view(), name='list'),
    path('studies/create/', views.study_create, name='create'),
    path('studies/<int:pk>/', views.study_detail, name='detail'),
    path('studies/<int:pk>/update/', views.study_update, name='update'),
    path('studies/<int:pk>/delete/', views.study_delete, name='delete'),
]
```

## 코드 설명

### 왜 `as_view()`를 붙이는가?

클래스는 그대로 URL에 연결할 수 없습니다.
Django가 요청을 처리할 수 있는 형태로 바꿔주는 메서드가 `as_view()`입니다.

이 부분은 클래스형 View를 처음 배울 때 아주 자주 헷갈립니다.

함수형 View:

```python
path('', views.study_list, name='list')
```

클래스형 View:

```python
path('', StudyListView.as_view(), name='list')
```

이 차이를 꼭 기억하세요.

## 지금 확인할 것

브라우저에서 `/`로 들어갔을 때,
이전과 똑같이 목록 페이지가 잘 보이면 성공입니다.

즉 사용자 입장에서는 똑같아 보이지만,
개발자 입장에서는 내부 구현이 바뀐 것입니다.

# 7. 클래스형 View와 함수형 View를 잠깐 비교해보기

여기서 한 번 비교하고 넘어가겠습니다.

| 방식        | 특징     | 장점         | 단점              |
| --------- | ------ | ---------- | --------------- |
| 함수형 View  | 가장 직관적 | 원리 이해 쉬움   | 반복 코드 많아질 수 있음  |
| 클래스형 View | 구조적    | 메서드별 분리 가능 | 처음엔 문법이 낯설 수 있음 |

즉 초보자 입장에서는 함수형 View가 더 쉽게 느껴질 수 있지만,
프로젝트가 커질수록 클래스형 View의 장점이 보이기 시작합니다.

하지만 Django는 여기서 한 단계 더 나아가,
반복되는 CRUD 구조 자체를 줄여주는 **Generic View**까지 제공합니다.

# 8. 제너릭 View는 왜 필요한가?

Django 프로젝트에서 정말 많이 반복되는 패턴이 있습니다.

* 목록 보기
* 상세 보기
* 생성
* 수정
* 삭제

이 구조는 게시판이든, 할 일 관리든, 공부 기록기든 거의 비슷합니다.

그래서 Django는 이런 반복을 줄이기 위해 미리 아래 클래스를 제공합니다.

* `ListView`
* `DetailView`
* `CreateView`
* `UpdateView`
* `DeleteView`

즉, 제너릭 View는 **“자주 쓰는 CRUD 패턴을 Django가 미리 제공해주는 구조”**라고 이해하면 좋습니다.

장점은 다음과 같습니다.

* 코드량 감소
* 구조가 더 Django답게 정리됨
* 유지보수와 확장이 쉬움

즉 3편의 중요한 목표 중 하나는,
Generic View를 보고 "**Django는 CRUD를 이런 식으로 정리하는구나**"를 느끼는 것입니다.

# 9. 제너릭 View용 파일 만들기

이제 제너릭 View 기준으로 CRUD를 다시 구성할 파일을 만듭니다.

```python title="study/views_generic.py"
from django.urls import reverse_lazy
from django.views.generic import CreateView, DeleteView, DetailView, ListView, UpdateView
from .forms import StudyForm
from .models import Study


class StudyListGenericView(ListView):
    model = Study
    template_name = 'study/study_list.html'
    context_object_name = 'page_obj'
    paginate_by = 5

    def get_context_data(self, **kwargs):
        context = super().get_context_data(**kwargs)
        studies = Study.objects.all()
        total_count = studies.count()
        total_minutes = sum(study.total_minutes for study in studies)
        context['total_count'] = total_count
        context['total_hours'] = total_minutes // 60
        context['remain_minutes'] = total_minutes % 60
        return context


class StudyDetailGenericView(DetailView):
    model = Study
    template_name = 'study/study_detail.html'
    context_object_name = 'study'


class StudyCreateGenericView(CreateView):
    model = Study
    form_class = StudyForm
    template_name = 'study/study_form.html'
    success_url = reverse_lazy('study:list')

    def get_context_data(self, **kwargs):
        context = super().get_context_data(**kwargs)
        context['page_title'] = '공부 기록 작성'
        return context


class StudyUpdateGenericView(UpdateView):
    model = Study
    form_class = StudyForm
    template_name = 'study/study_form.html'

    def get_success_url(self):
        return reverse_lazy('study:detail', kwargs={'pk': self.object.pk})

    def get_context_data(self, **kwargs):
        context = super().get_context_data(**kwargs)
        context['page_title'] = '공부 기록 수정'
        return context


class StudyDeleteGenericView(DeleteView):
    model = Study
    template_name = 'study/study_confirm_delete.html'
    success_url = reverse_lazy('study:list')
    context_object_name = 'study'
```

## 코드 설명

### `ListView`

목록 조회 전용 View입니다.
기본적으로 queryset을 가져와 템플릿에 전달합니다.

### `DetailView`

하나의 객체를 자세히 보여주는 View입니다.

### `CreateView`

생성 폼을 보여주고, POST 요청 시 저장까지 처리해줍니다.
즉 “폼 보여주기 + 저장” 패턴이 내장되어 있습니다.

### `UpdateView`

수정용 View입니다.
기존 객체를 찾아서 수정 폼에 연결해줍니다.

### `DeleteView`

삭제 확인과 삭제 처리를 담당합니다.

### 왜 `reverse_lazy()`를 쓰는가?

클래스 속성 단계에서 URL 이름을 역참조할 때는 `reverse_lazy()`를 자주 사용합니다.
이 부분은 제너릭 View나 클래스형 View에서 자주 보게 되는 문법입니다.

### 왜 `get_context_data()`를 오버라이드하나?

기본적으로 제너릭 View는 필요한 기본 데이터만 넘깁니다.
하지만 이번 실습에서는:

* 전체 기록 수
* 누적 공부 시간
* 페이지 제목

처럼 추가로 넘겨야 하는 데이터가 있습니다.

그래서 `get_context_data()`를 사용해 context를 확장합니다.

# 10. 제너릭 View URL 연결로 바꾸기

# 11. 함수형 / 클래스형 / 제너릭 View 총정리

여기서 한 번 정리하고 넘어가겠습니다.

| 방식        | 특징       | 장점                    | 단점             |
| --------- | -------- | --------------------- | -------------- |
| 함수형 View  | 가장 직관적   | 원리 이해 쉬움              | 반복 코드 많아질 수 있음 |
| 클래스형 View | 구조적      | 메서드별 분리 가능            | 문법이 낯설 수 있음    |
| 제너릭 View  | CRUD에 강함 | 코드량 감소, Django 패턴에 적합 | 내부 동작 이해 필요    |

초보자 학습 순서는 보통 아래가 좋습니다.

```
함수형 View 먼저 이해
→ 클래스형 View 문법 익히기
→ 제너릭 View로 Django 패턴 익히기
```

이번 교안은 바로 이 순서를 따라가고 있습니다.

# 12. 이제 API는 왜 필요한가?

지금까지는 브라우저 화면을 기준으로 기능을 만들었습니다.
즉 HTML 템플릿과 Django View가 직접 연결되어 있었습니다.

하지만 나중에는 아래 같은 상황이 생깁니다.

* React / Next.js 프론트엔드에서 Django 데이터를 쓰고 싶다.
* 모바일 앱에서 같은 데이터를 쓰고 싶다.
* JavaScript로 비동기 요청을 보내고 싶다.
* 프론트와 백엔드를 분리하고 싶다.

이럴 때 필요한 것이 **API**입니다.

즉 HTML 응답이 아니라,
JSON 형태로 데이터를 주고받는 구조가 필요해집니다.

이번 교안에서는 그 첫 단계로 **Django REST Framework(DRF)** 를 붙여보겠습니다.

# 14. Serializer는 무엇인가?

DRF를 처음 배울 때 가장 먼저 익혀야 하는 개념 중 하나가 **Serializer**입니다.

쉽게 말하면 Serializer는 아래 역할을 합니다.

```
Django 모델 객체
↔ Serializer
↔ JSON 데이터
```

즉:

* 모델 객체를 JSON으로 바꿔주고
* JSON 입력값을 검증해서 모델로 저장할 수 있게 도와줍니다.

Form이 HTML 폼 입력을 다루는 느낌이라면,
Serializer는 **JSON 데이터를 다루는 Form 비슷한 역할**이라고 생각해도 좋습니다.

---

# 15. Serializer 작성

이제 `Study` 모델용 Serializer를 만듭니다.

```python title="study/serializers.py"
from rest_framework import serializers
from .models import Study


class StudySerializer(serializers.ModelSerializer):
    total_minutes = serializers.ReadOnlyField()
    category_display = serializers.CharField(source='get_category_display', read_only=True)

    class Meta:
        model = Study
        fields = [
            'id',
            'title',
            'category',
            'category_display',
            'study_date',
            'hours',
            'minutes',
            'total_minutes',
            'memo',
            'created_at',
            'updated_at',
        ]
```

## 코드 설명

### `ModelSerializer`

Django 모델과 연결된 Serializer입니다.
ModelForm처럼 모델 구조를 바탕으로 빠르게 만들 수 있습니다.

### `total_minutes = serializers.ReadOnlyField()`

모델의 계산용 속성을 읽기 전용으로 API 응답에 포함했습니다.

즉 API 사용자 입장에서는 따로 계산하지 않아도,
총 공부 시간이 바로 JSON에 포함되어 내려옵니다.

### `category_display`

저장값(`django`)뿐 아니라 사람이 보기 좋은 값(`Django`)도 같이 내려줍니다.
이렇게 하면 API 응답이 더 읽기 쉬워집니다.

즉 Serializer는 단순 변환 도구를 넘어서,
**API 응답을 더 친절하게 가공하는 역할도 할 수 있습니다.**

# 16. ViewSet은 무엇인가?

브라우저 쪽에서 Generic View가 CRUD를 크게 줄여줬다면,
DRF 쪽에서는 **ViewSet**이 비슷한 역할을 합니다.

ViewSet은 여러 API 동작을 하나의 클래스에 묶어서 다루는 방식입니다.

예를 들어 아래 같은 기능이 하나의 ViewSet 안에 들어갈 수 있습니다.

* 목록 조회
* 상세 조회
* 생성
* 수정
* 삭제

즉 ViewSet은 **API용 CRUD 묶음 클래스**라고 이해하면 좋습니다.

---

# 17. ViewSet 작성

이제 실제 ViewSet을 작성합니다.

```python title="study/api_views.py"
from rest_framework import viewsets
from .models import Study
from .serializers import StudySerializer


class StudyViewSet(viewsets.ModelViewSet):
    queryset = Study.objects.all()
    serializer_class = StudySerializer
```

## 코드 설명

### `ModelViewSet`

`ModelViewSet`은 매우 강력합니다.
하나의 클래스만으로 아래 기능을 거의 모두 제공합니다.

* 목록 조회
* 상세 조회
* 생성
* 수정
* 삭제

즉 브라우저 쪽에서 Generic View로 CRUD를 줄였듯이,
API 쪽에서는 `ModelViewSet`이 같은 역할을 해줍니다.

### `queryset`

어떤 데이터를 대상으로 할지 정합니다.
여기서는 `Study.objects.all()`이므로 모든 공부기록을 대상으로 합니다.

### `serializer_class`

어떤 Serializer를 사용할지 정합니다.
즉 이 ViewSet은 `StudySerializer`를 통해 JSON을 주고받게 됩니다.

# 18. Router는 왜 필요한가?

이제 ViewSet이 준비되었지만,
URL까지 일일이 손으로 다 연결하면 오히려 번거롭습니다.

예를 들어 API는 보통 아래 URL이 필요합니다.

```
GET    /api/studies/
POST   /api/studies/
GET    /api/studies/1/
PUT    /api/studies/1/
PATCH  /api/studies/1/
DELETE /api/studies/1/
```

이걸 전부 직접 path로 작성할 수도 있지만,
DRF는 이를 자동으로 만들어주는 **Router**를 제공합니다.

즉 Router는 **ViewSet용 URL 자동 생성기**라고 이해하면 됩니다.

---

# 19. Router 연결용 api_urls.py 작성

이제 API 전용 URL 파일을 만듭니다.

```python title="study/api_urls.py"
from rest_framework.routers import DefaultRouter
from .api_views import StudyViewSet

router = DefaultRouter()
router.register('studies', StudyViewSet, basename='study')

urlpatterns = router.urls
```

## 코드 설명

### `DefaultRouter()`

기본 Router를 생성합니다.

### `router.register('studies', StudyViewSet, basename='study')`

`StudyViewSet`을 `studies` 경로에 연결합니다.
즉 이제 `/api/studies/` 계열 URL이 자동으로 만들어집니다.

### `urlpatterns = router.urls`

Router가 만들어준 URL 목록을 그대로 URL 패턴으로 사용합니다.

즉 DRF에서는 아래 흐름을 기억하는 것이 중요합니다.

```
Serializer
→ ViewSet
→ Router
```

### PUT 과 PATCH 차이

PUT은 전체 수정입니다.
즉 기존 객체를 일부만 고치는 개념보다는, 새 데이터로 전체를 교체하는 방식에 가깝죠.

그래서 PUT 요청을 보낼 때는 보통 해당 객체의 전체 필드를 함께 보내야 합니다.

예를 들어 공부기록이 아래처럼 있을 때,

```json
{
  "id": 1,
  "title": "DRF ViewSet 복습",
  "category": "django",
  "study_date": "2026-03-13",
  "hours": 1,
  "minutes": 40,
  "memo": "Serializer와 Router 흐름을 다시 정리했다."
}
```

PUT은 이 객체를 통째로 새 상태로 바꾸는 요청으로 봅니다.
그래서 minutes만 보내면, 서버 입장에서는 나머지 필드가 빠진 것으로 판단할 수 있습니다.

> 그럼 위 사진에서는 왜 title, study_date만 필수라고 뜨나요?

* title, study_date는 모델과 Serializer 기준에서 필수 필드이기 때문입니다. study/models.py 코드를 봐볼까요?

```python title="study/models.py"
from django.db import models

class Study(models.Model):
    CATEGORY_CHOICES = [
        ('python', 'Python'),
        ('django', 'Django'),
        ('algorithm', '알고리즘'),
        ('database', 'Database'),
        ('etc', '기타'),
    ]

    title = models.CharField(max_length=200, verbose_name='공부 제목')
    category = models.CharField(max_length=30, choices=CATEGORY_CHOICES, default='django', verbose_name='카테고리')
    study_date = models.DateField(verbose_name='공부 날짜')
    hours = models.PositiveIntegerField(default=0, verbose_name='시간')
    minutes = models.PositiveIntegerField(default=0, verbose_name='분')
    memo = models.TextField(blank=True, verbose_name='메모')
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['-study_date', '-created_at']
        verbose_name = '공부 기록'
        verbose_name_plural = '공부 기록들'

    def __str__(self):
        return f'{self.title} ({self.study_date})'

    @property
    def total_minutes(self):
        return self.hours * 60 + self.minutes
```

위 코드 중 아래 두 줄을 살펴봅시다.

```python
title = models.CharField(max_length=200, verbose_name='공부 제목')
study_date = models.DateField(verbose_name='공부 날짜')
```

여기에는:

- blank=True 없음

- null=True 없음

- default도 없음

즉 Django 입장에서는 반드시 값이 있어야 하는 필드로 봅니다.

그리고 ModelSerializer는 이 모델 정보를 바탕으로
Serializer 필드를 자동 생성하면서 required=True에 해당하는 성격으로 처리합니다.

> 그럼 memo는요?

```python
memo = models.TextField(blank=True, verbose_name='메모')
```

blank=True가 있으면, 
폼이나 Serializer 검증에서 “비워도 됨”으로 해석될 수 있습니다.

즉 memo는 선택 입력처럼 동작하고,
title, study_date는 필수 입력처럼 동작하는 이유가 여기서 나옵니다.

# 25. 이번 교안에서 핵심 흐름

이번 교안의 핵심을 한 줄 흐름으로 정리하면 아래와 같습니다.

```
함수형 View로 CRUD 이해
→ 클래스형 View로 구조화 이해
→ 제너릭 View로 Django 패턴 정리
→ DRF Serializer
→ ViewSet
→ Router
→ API 테스트
```

즉 이번 편의 핵심은 **브라우저 기반 CRUD를 넘어 Django 구조 확장과 API 확장까지 경험하는 것**입니다.

---

# 26. 자주 하는 실수 정리

## 1) 클래스형 View를 URL에 연결하면서 `as_view()`를 빼먹음

이 경우 클래스형 View가 정상적으로 동작하지 않습니다.

## 2) 제너릭 View로 바꿨는데 템플릿 변수명을 그대로 두지 않음

예를 들어 목록 페이지에서 `page_obj`나 `study` 같은 이름을 일관되게 맞춰줘야 합니다.

## 3) `reverse_lazy()` 대신 잘못된 URL 처리

제너릭 View의 `success_url`이나 클래스 속성에서 URL 역참조 시 문법을 혼동할 수 있습니다.

## 4) DRF 설치 후 `INSTALLED_APPS`에 `rest_framework` 등록을 안 함

이 경우 DRF 관련 기능이 정상적으로 동작하지 않습니다.

## 5) Serializer는 만들었는데 ViewSet에서 연결하지 않음

그러면 JSON 응답 구조가 준비되지 않습니다.

## 6) Router는 만들었는데 `config/urls.py`에서 include하지 않음

`/api/studies/`에 접속해도 페이지가 열리지 않습니다.

## 7) 브라우저 CRUD와 API CRUD를 서로 완전히 다른 데이터라고 착각함

둘 다 결국 같은 `Study` 모델 / 같은 DB를 바라보고 있다는 점을 기억해야 합니다.

# 28. 결론

이번 교안에서 여러분이 한 것은 단순히 DRF 설치를 해본 것이 아닙니다.
실제로 더 중요하게 익힌 것은 아래 두 가지입니다.

## 1) Django 구조를 더 Django답게 정리하는 방법

* 함수형 View
* 클래스형 View
* 제너릭 View

이 흐름을 통해, Django가 왜 이런 구조를 제공하는지 이해하게 되었습니다.

## 2) 같은 데이터를 API로 확장하는 방법

* Serializer
* ViewSet
* Router
* API 테스트

를 통해 브라우저 기반 서비스가 API 기반 구조로도 확장될 수 있음을 확인했습니다.

즉 이제 여러분은 단순히 Django 페이지 하나를 만드는 수준을 넘어,
아래 전체 흐름을 한 번 완주한 것입니다.

```
프로젝트 시작
→ 모델 설계
→ 브라우저 기반 CRUD
→ View 구조 확장
→ DRF API 확장
```

이 정도까지 해보면 Django의 큰 뼈대는 한 번 잡혔다고 볼 수 있습니다.

그리고 정말 중요한 것은,
이제부터 이 흐름을 다른 주제로 다시 반복해보는 것입니다.

예를 들어:

* 독서 기록기
* 운동 기록기
* 감정 일기장
* 지출 기록기

같은 주제로 다시 만들어보면 Django 구조가 훨씬 더 빠르게 익숙해질 것입니다.

이번 교안 주제인 "공부시간 기록기" 1~3편을 모두 마치면,
Django의 가장 기본적인 실습 사이클은 한 번 제대로 경험한 것입니다.
