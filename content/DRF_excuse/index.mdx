# DRF + 로컬 LLM으로 AI 핑계 생성기 만들기

이번 교안에서는 **Django REST Framework(DRF)** 와 **로컬 LLM**을 함께 사용하여,
**AI 핑계 생성 서비스**를 만들어 봅니다.

이번 프로젝트의 핵심은 단순히 API 하나를 띄우는 것이 아닙니다.
아래 흐름을 **하나의 서비스 구조로 자연스럽게 연결**해 보는 것이 중요합니다.

```
사용자 입력 (situation, tone)
→ DRF API 요청
→ 로컬 LLM(Qwen2.5-1.5B-Instruct)이 핑계 생성
→ SQLite DB 저장
→ 목록 / 상세 / 랜덤 조회
→ HTML 페이지에서 결과 확인
```

즉, 이번 교안을 통해 아래 내용을 학습할 수 있습니다!

* DRF 기본 구조
* Serializer
* Generic View / APIView
* 커스텀 API 엔드포인트
* Service Layer 분리
* SQLite 저장
* HTML + JavaScript 연동
* GET / POST / PUT / DELETE 실습
* 로컬 LLM을 웹 서비스에 연결하는 흐름

또한 “API가 실제 서비스에서 어떻게 사용되는지”까지 이해할 수 있도록,
**각 기능을 확인할 수 있는 HTML 페이지도 함께 구현**합니다.

> 이번 교안은 **SQLite**를 기준으로 진행합니다.
> **PostgreSQL로 변경하는 방법은 추후 교안**에서 이어서 다룰 예정입니다.

* 로컬 LLM을 API에 연결하는 방법
* API와 HTML 화면을 연결하는 방법
* 기존 CRUD 서비스 위에 AI 생성 기능을 추가하는 방법

# 2. Qwen2.5-1.5B-Instruct 모델을 사용합니다.

## 왜 이 모델이 더 적합할까요?

* 0.5B보다 더 자연스러운 문장 생성 가능
* 한국어 처리 품질이 상대적으로 안정적
* 짧은 상황 기반 생성에서 더 나은 결과 기대 가능
* 여전히 로컬 실습용으로 시도 가능한 크기

물론 이것도 아주 큰 모델은 아니기 때문에,
프롬프트 설계와 후처리가 여전히 중요합니다.

즉,
**모델만 바꾸면 끝나는 것이 아니라 프롬프트와 후처리도 함께 개선해야 합니다.**

## 왜 `services.py`를 따로 둘까요?

AI 관련 로직을 View 안에 직접 넣으면 코드가 금방 복잡해집니다.
그래서 아래처럼 역할을 나눕니다.

* View: 요청 받고 응답 반환
* Service: 실제 AI 생성 처리

이렇게 분리하면

* 코드가 깔끔해지고
* 유지보수가 쉬워지고
* 나중에 모델을 바꾸기도 편해집니다.

# 11. 모델 설계

이제 실제 데이터를 저장할 모델을 만듭니다.

```python title="excuses/models.py"
from django.db import models


class Excuse(models.Model):
    situation = models.CharField(max_length=100)
    tone = models.CharField(max_length=50)
    content = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    def __str__(self):
        return self.content
```

## 코드 설명

### `situation = models.CharField(max_length=100)`

사용자가 어떤 상황에서 사용할 핑계를 만들고 싶은지 저장합니다.
예: `지각`, `결석`, `약속`, `과제 미제출`

### `tone = models.CharField(max_length=50)`

핑계의 말투나 분위기를 저장합니다.
예: `진지함`, `가벼움`, `유머`, `죄송함`, `당황`

> 이번 교안에서는 tone을 **한국어로 통일**하는 것을 권장합니다.
> 영어와 한국어를 섞어 쓰면 모델 출력이 더 불안정해질 수 있습니다.

### `content = models.TextField()`

AI가 실제로 생성한 핑계 문장을 저장합니다.
문장 길이가 유동적일 수 있으므로 `TextField`를 사용합니다.

### `created_at = models.DateTimeField(auto_now_add=True)`

데이터가 **처음 생성될 때의 시간**을 자동으로 저장합니다.

### `updated_at = models.DateTimeField(auto_now=True)`

데이터가 **수정될 때마다 최신 시간**으로 자동 갱신됩니다.

### `def __str__(self):`

Django Admin이나 쉘에서 객체를 볼 때 사람이 읽기 쉬운 문자열로 보여 주는 부분입니다.

## 코드 설명

### `admin.site.register(Excuse)`

이 한 줄로 `Excuse` 모델이 Django Admin에 등록됩니다.

즉,
브라우저에서 `/admin/`으로 들어갔을 때

실습 중에는 API로만 테스트해도 되지만,
관리자 페이지가 있으면 DB에 어떤 값이 들어갔는지 훨씬 쉽게 확인할 수 있습니다.

### 관리자 계정 생성은 아직 하지 않습니다

`createsuperuser`는 **마이그레이션이 끝난 뒤** 실행해야 합니다.

왜냐하면 관리자 계정도 결국 `auth_user` 같은 기본 Django 테이블이 준비되어 있어야 만들 수 있기 때문입니다.
또한 이번 실습에서는 `config/urls.py`, `excuses/urls.py`, `views.py`까지 먼저 완성한 뒤에
명령어를 실행하는 흐름이 더 안전합니다.

# 13. Serializer 작성

DRF에서 Serializer는 아주 중요합니다.

역할은 크게 두 가지입니다.

> 1. Python 객체를 JSON으로 바꿔 줍니다
> 2. 요청으로 들어온 데이터를 검증해 줍니다

serializers.py 생성해주세요.

```python title="excuses/serializers.py"
from rest_framework import serializers
from .models import Excuse


class ExcuseSerializer(serializers.ModelSerializer):
    class Meta:
        model = Excuse
        fields = '__all__'


class ExcuseGenerateSerializer(serializers.Serializer):
    situation = serializers.CharField()
    tone = serializers.CharField()
```

## 코드 설명

### `ExcuseSerializer`

이 Serializer는 `Excuse` 모델 전체를 다룹니다.

즉,

* 목록 조회 응답
* 상세 조회 응답
* 수정 응답
* 삭제 전 확인

등에 사용됩니다.

`ModelSerializer`를 사용하면 모델 필드를 자동으로 반영할 수 있어서 편리합니다.

### `ExcuseGenerateSerializer`

이 Serializer는 AI 생성 요청에서 들어오는 입력값만 검증합니다.

즉,

```json
{
  "situation": "지각",
  "tone": "진지함"
}
```

이런 요청에서 `situation`, `tone`이 제대로 들어왔는지 확인합니다.

## 왜 Serializer를 2개로 나누나요?

실무에서도 **입력용 Serializer**와 **출력용 Serializer**를 분리하는 경우가 많습니다.
이렇게 하면 역할이 명확해지고 유지보수가 쉬워집니다.

# 14. AI 서비스 로직

이제 가장 중요한 부분입니다.
로컬 LLM을 사용해 핑계를 생성해 보겠습니다.

이번 버전에서는 아래 사항을 반영합니다.

* `Qwen2.5-1.5B-Instruct` 모델 사용
* 프롬프트를 **완전히 한국어 기반**으로 작성
* 결과에서 프롬프트가 섞여 나오는 문제를 줄이기 위해 `return_full_text=False` 사용
* 반복 / 영어 응답 / 설명문 출력을 줄이기 위해 파라미터 조정
* 간단한 후처리 추가

services.py 생성해주세요.

```python title="excuses/services.py"
from transformers import pipeline


generator = pipeline(
    "text-generation",
    model="Qwen/Qwen2.5-1.5B-Instruct"
)


def generate_excuse(situation, tone):

    prompt = f"""
다음 상황에 대한 자연스러운 핑계를 한 문장으로 만들어라.

상황: {situation}
톤: {tone}

조건:
- 반드시 한국어 한 문장
- 설명하지 말 것
- 상황에 맞는 핑계만 출력

핑계:
"""

    result = generator(
        prompt,
        do_sample=True,
        max_new_tokens=100,
        temperature=0.7,
        top_p=0.9,
        repetition_penalty=1.1,
        return_full_text=False,
        truncation=True,
    )

    text = result[0]["generated_text"].strip()

    if "핑계:" in text:
        text = text.split("핑계:")[-1].strip()

    text = text.split("\n")[0]

    return text
```

## 코드 설명

- `pipeline("text-generation", model="Qwen/Qwen2.5-1.5B-Instruct")`

Hugging Face의 `pipeline`을 사용해 텍스트 생성용 모델을 불러옵니다.

- `prompt = ...`

프롬프트는 모델에게 무엇을 생성해야 하는지 알려 주는 지시문입니다.

이번에는 프롬프트를 **영어가 아니라 한국어 중심**으로 구성했습니다.
그 이유는 모델이 프롬프트 언어를 따라가는 경향이 있기 때문입니다.

- `핑계:`

이 부분은 모델에게 “이제 실제 답변을 시작해라”는 역할을 합니다.
다만 모델이 `핑계:`를 함께 출력할 수 있기 때문에,
아래 후처리에서 제거합니다.

- `do_sample=True`

같은 입력이 들어와도 약간씩 다른 결과를 만들 수 있게 해 줍니다.
즉, 생성형 서비스다운 느낌을 주기 위해 필요한 옵션입니다.

- `max_new_tokens=100`

생성 길이를 제한하는 옵션입니다.
핑계는 보통 짧은 문장으로 충분하지만,
너무 작게 잡으면 중간에 잘릴 수 있어서 100으로 설정했습니다.

- `temperature=0.7`

생성 결과를 너무 들쭉날쭉하지 않으면서도,
조금은 다양하게 만들기 위한 설정입니다.

- `top_p=0.9`

너무 엉뚱한 단어 선택을 줄이고,
상대적으로 자연스러운 후보들 안에서 문장을 생성하게 돕습니다.

- `repetition_penalty=1.1`

같은 단어나 구절이 반복되는 현상을 줄이기 위한 설정입니다.

- `return_full_text=False`

이 옵션이 매우 중요합니다.
기본적으로 text-generation은 프롬프트 + 결과를 함께 반환하는 경우가 있는데,
이 옵션을 사용하면 **생성 결과만 반환**하게 할 수 있습니다.

즉,
화면에 `상황: 지각`, `톤: 진지함` 같은 문장이 그대로 노출되는 문제를 줄일 수 있습니다.

- `truncation=True`

프롬프트가 길어질 경우 잘라서 처리하도록 하는 옵션입니다.

- `text = result[0]["generated_text"].strip()`

모델이 생성한 최종 텍스트를 꺼내오는 부분입니다.

- `if "핑계:" in text:`

모델이 `핑계:`까지 같이 출력했을 경우,
실제 문장만 남기기 위해 제거합니다.

- `text = text.split("\n")[0]`

모델이 여러 줄을 출력할 경우,
첫 줄만 사용하도록 정리합니다.

# 15. DRF View 작성

이제 DRF의 핵심인 View를 작성해 보겠습니다.

이번 교안에서는 아래 뷰를 사용합니다.

* `ListAPIView`
* `RetrieveUpdateDestroyAPIView`
* `GenericAPIView`
* `TemplateView`

```python title="excuses/views.py"
import random

from django.views.generic import TemplateView
from rest_framework.generics import ListAPIView
from rest_framework.generics import RetrieveUpdateDestroyAPIView
from rest_framework.generics import GenericAPIView
from rest_framework.response import Response

from .models import Excuse
from .serializers import ExcuseSerializer
from .serializers import ExcuseGenerateSerializer
from .services import generate_excuse


# HTML PAGE VIEWS
class HomePageView(TemplateView):
    template_name = 'excuses/home.html'


class ExcuseListPageView(TemplateView):
    template_name = 'excuses/excuse_list.html'


class ExcuseDetailPageView(TemplateView):
    template_name = 'excuses/excuse_detail.html'


class ExcuseRandomPageView(TemplateView):
    template_name = 'excuses/excuse_random.html'


class ExcuseGeneratePageView(TemplateView):
    template_name = 'excuses/excuse_generate.html'


# API VIEWS
class ExcuseListView(ListAPIView):
    queryset = Excuse.objects.all().order_by('-id')
    serializer_class = ExcuseSerializer


class ExcuseDetailView(RetrieveUpdateDestroyAPIView):
    queryset = Excuse.objects.all()
    serializer_class = ExcuseSerializer


class ExcuseRandomView(GenericAPIView):
    serializer_class = ExcuseSerializer

    def get(self, request):
        excuses = Excuse.objects.all()

        if not excuses.exists():
            return Response({"message": "데이터가 없습니다"})

        excuse = random.choice(excuses)
        serializer = ExcuseSerializer(excuse)
        return Response(serializer.data)


class ExcuseGenerateView(GenericAPIView):
    serializer_class = ExcuseGenerateSerializer

    def post(self, request):
        serializer = self.get_serializer(data=request.data)
        serializer.is_valid(raise_exception=True)

        situation = serializer.validated_data['situation']
        tone = serializer.validated_data['tone']

        content = generate_excuse(situation, tone)

        excuse = Excuse.objects.create(
            situation=situation,
            tone=tone,
            content=content,
        )

        result = ExcuseSerializer(excuse)
        return Response(result.data)
```

## 코드 설명

### `TemplateView`

HTML 페이지를 보여 주기 위한 Django 기본 뷰입니다.
이번 실습에서는 API뿐 아니라 브라우저 화면도 함께 확인하기 위해 사용합니다.

즉,

* `HomePageView`
* `ExcuseListPageView`
* `ExcuseDetailPageView`
* `ExcuseRandomPageView`
* `ExcuseGeneratePageView`

은 모두 “화면 렌더링용” 뷰입니다.

---

### `class ExcuseListView(ListAPIView)`

이 뷰는 **전체 목록 조회 전용 API**입니다.

#### 왜 `ListAPIView`를 쓰나요?

목록 조회만 할 때 가장 간단하고,
불필요한 코드를 많이 줄여 줍니다.

#### `queryset = Excuse.objects.all().order_by('-id')`

모든 Excuse 데이터를 가져오되,
최신 데이터가 먼저 오도록 정렬합니다.

#### `serializer_class = ExcuseSerializer`

조회된 Excuse 객체들을 JSON으로 바꿔서 응답하도록 설정합니다.

---

### `class ExcuseDetailView(RetrieveUpdateDestroyAPIView)`

이 뷰는 **상세 조회 / 수정 / 삭제**를 한 번에 담당합니다.

#### 왜 이 뷰를 쓰나요?

이 클래스 하나로 아래 기능을 모두 처리할 수 있습니다.

* GET `/api/excuses/<id>/`
* PUT `/api/excuses/<id>/`
* DELETE `/api/excuses/<id>/`

즉, CRUD 중 “단일 객체 처리”에 매우 편리합니다.

---

### `class ExcuseRandomView(GenericAPIView)`

이 뷰는 랜덤 핑계를 가져오는 **커스텀 API**입니다.

#### 왜 `GenericAPIView`를 쓰나요?

랜덤 조회는 기본 CRUD 패턴에 딱 맞지 않습니다.
그래서 자유롭게 동작을 커스터마이징할 수 있는 `GenericAPIView`가 적합합니다.

#### `excuses.exists()`

DB에 데이터가 하나도 없으면 랜덤 추출이 불가능합니다.
그래서 먼저 데이터 존재 여부를 확인합니다.

#### `random.choice(excuses)`

저장된 핑계 중 하나를 무작위로 선택합니다.

---

### `class ExcuseGenerateView(GenericAPIView)`

이 뷰는 AI 핑계 생성을 담당하는 핵심 API입니다.

#### `serializer = self.get_serializer(data=request.data)`

사용자가 보낸 JSON 요청을 Serializer에 넣어 검증 준비를 합니다.

#### `serializer.is_valid(raise_exception=True)`

입력값이 유효한지 확인합니다.
잘못된 값이면 자동으로 에러 응답을 반환합니다.

#### `validated_data`

검증을 통과한 안전한 데이터만 꺼내옵니다.

#### `content = generate_excuse(situation, tone)`

서비스 레이어의 AI 함수로 실제 핑계를 생성합니다.

#### `Excuse.objects.create(...)`

생성된 결과를 DB에 저장합니다.

#### `result = ExcuseSerializer(excuse)`

저장된 모델 객체를 JSON으로 바꿔 응답합니다.

즉,
이 API는 아래 흐름으로 동작합니다.

```
요청 받기
→ 입력 검증
→ LLM 생성
→ DB 저장
→ JSON 응답
```

```html title="excuses/templates/excuses/excuse_detail.html"
    `;

    document.getElementById('situation').value = data.situation;
    document.getElementById('tone').value = data.tone;
    document.getElementById('content').value = data.content;
}

async function updateExcuse() {
    const response = await fetch(`/api/excuses/` + excuseId + `/`, {
        method: 'PUT',
        headers: {
            'Content-Type': 'application/json'
        },
        body: JSON.stringify({
            situation: document.getElementById('situation').value,
            tone: document.getElementById('tone').value,
            content: document.getElementById('content').value
        })
    });

    await response.json();
    alert('수정 완료');
    loadDetail();
}

async function deleteExcuse() {
    await fetch(`/api/excuses/` + excuseId + `/`, {
        method: 'DELETE'
    });

    alert('삭제 완료');
    window.location.href = '/excuses/';
}

loadDetail();
</script>
{% endblock %}
```

## 코드 설명

### `window.location.pathname.split('/')`

현재 URL에서 `id` 값을 추출하는 코드입니다.
예를 들어 `/excuses/3/page/`라면 `3`을 꺼내서 API 호출에 사용합니다.

### `fetch(`/api/excuses/` + excuseId + `/`)`

특정 id에 해당하는 핑계 상세 데이터를 요청합니다.

### PUT 버튼

수정 버튼을 누르면 아래 정보가 다시 서버로 전송됩니다.

* situation
* tone
* content

즉, 학생들은 **브라우저에서 직접 PUT 요청**을 경험할 수 있습니다.

### DELETE 버튼

삭제 버튼을 누르면 해당 데이터를 삭제한 뒤,
목록 페이지로 다시 이동합니다.

즉, 학생들은 **브라우저에서 직접 DELETE 요청**도 경험할 수 있습니다.

# 19. 마이그레이션 + 관리자 계정 생성

이제 **실제로 `manage.py` 명령어를 실행해도 되는 시점**입니다.

여기까지 오면 아래 파일들이 준비되어 있어야 합니다.

* `config/settings.py`
* `config/urls.py`
* `excuses/models.py`
* `excuses/admin.py`
* `excuses/serializers.py`
* `excuses/services.py`
* `excuses/views.py`
* `excuses/urls.py`

즉, Django가 프로젝트를 읽는 데 필요한 핵심 파일이 먼저 준비된 뒤에
마이그레이션과 관리자 계정 생성을 진행하는 것이 안전합니다.

이 순서를 지키면  
`No module named ...`, `ModuleNotFoundError`, `include('excuses.urls')` 관련 오류,
그리고 `createsuperuser` 실행 시 발생하는 URL / View import 오류를 훨씬 줄일 수 있습니다.

## 19-1. 마이그레이션

모델을 만들었으면 실제 DB에 테이블을 반영해야 합니다.

```bash
python manage.py makemigrations
python manage.py migrate
```

### 설명

#### `makemigrations`

모델 변경사항을 Django가 이해할 수 있는 “설계 변경 기록”으로 만듭니다.

#### `migrate`

그 변경 기록을 실제 데이터베이스(SQLite)에 적용합니다.

즉,

```
models.py 작성
→ makemigrations
→ migrate
→ DB 테이블 생성
```

이 순서로 이해하면 됩니다.

또한 `migrate`를 실행하면
우리가 만든 `Excuse` 테이블뿐 아니라
Django 기본 인증/관리자 관련 테이블도 함께 준비됩니다.

그래서 **`createsuperuser`는 반드시 `migrate` 이후**에 실행해야 합니다.

## 19-2. 관리자 계정 생성

이제 관리자 페이지 로그인을 위한 계정을 생성합니다.

```bash
python manage.py createsuperuser
```

실행하면 아래 항목을 차례대로 입력합니다.

* Username
* Email address
* Password

계정 생성이 끝나면 나중에 `/admin/` 으로 접속해서 로그인할 수 있습니다.

### 왜 여기서 만드는 것이 좋을까요?

많은 초보자들이 `admin.py`를 만들자마자 `createsuperuser`를 실행하는데,
실제로는 이 시점이 아직 안전하지 않을 수 있습니다.

왜냐하면 Django는 명령어 실행 시 프로젝트 전체 설정과 URL, View import까지 함께 확인하는 경우가 많기 때문입니다.

그래서 이번 교안처럼 아래 순서를 추천합니다.

```
모델 / Serializer / Service / View / URL 작성 완료
→ makemigrations
→ migrate
→ createsuperuser
→ runserver
```

## 21-2. 핑계 상세 조회

### 요청 정보

* **Method**: `GET`
* **URL**: `/api/excuses/<id>/`

### 어떻게 요청하나요?

예를 들어 id가 1인 핑계를 보고 싶다면 아래처럼 요청합니다.

```text

```

또는 목록 페이지에서 상세 링크를 눌러 확인할 수 있습니다.

### 예시 응답

```json
{
  "id": 1,
  "situation": "지각",
  "tone": "진지함",
  "content": "예상보다 이동 시간이 길어져 조금 늦었습니다.",
  "created_at": "2026-03-09T09:30:00Z",
  "updated_at": "2026-03-09T09:30:00Z"
}
```

### 응답값 의미

전체 조회와 동일하지만,
여기서는 특정 핑계 한 개만 객체(`{}`) 형태로 옵니다.

### 이 요청에서 확인할 것

* 배열이 아니라 객체 하나가 오는지
* URL의 id와 응답 id가 일치하는지
* 존재하지 않는 id를 요청했을 때 404가 나오는지

## 21-5. 핑계 수정

### 요청 정보

* **Method**: `PUT`
* **URL**: `/api/excuses/<id>/`

### 어떻게 요청하나요?

PUT 요청은 브라우저 주소창에서 직접 보내기 어렵기 때문에,
Insomnia 또는 Postman 사용을 권장합니다.

하지만 이번 교안에서는 **상세 페이지의 수정 버튼**으로도 PUT 요청을 테스트할 수 있습니다.

예를 들어 id가 1인 핑계를 수정하려면 아래처럼 요청합니다.

```text
PUT http://127.0.0.1:8000/api/excuses/1/
```

### 요청 Body 예시

```json
{
  "situation": "지각",
  "tone": "진지함",
  "content": "예상보다 이동 시간이 길어져 도착이 조금 늦어졌습니다."
}
```

### 요청값 의미

PUT은 **전체 수정** 개념입니다.
즉, 필요한 필드를 모두 다시 보내는 것이 안전합니다.

### 예시 응답

```json
{
  "id": 1,
  "situation": "지각",
  "tone": "진지함",
  "content": "예상보다 이동 시간이 길어져 도착이 조금 늦어졌습니다.",
  "created_at": "2026-03-09T09:30:00Z",
  "updated_at": "2026-03-09T11:00:00Z"
}
```

### 이 요청에서 확인할 것

* `content`가 실제로 수정되었는지
* `updated_at`이 변경되었는지
* 상세 조회에서 수정된 내용이 보이는지
* HTML 상세 페이지의 수정 버튼으로도 동작하는지

---

## 21-6. 핑계 삭제

### 요청 정보

* **Method**: `DELETE`
* **URL**: `/api/excuses/<id>/`

### 어떻게 요청하나요?

이 요청도 보통 Insomnia / Postman에서 테스트합니다.

하지만 이번 교안에서는 **상세 페이지의 삭제 버튼**으로도 DELETE 요청을 테스트할 수 있습니다.

예시:

```text
DELETE http://127.0.0.1:8000/api/excuses/1/
```

### 응답 의미

삭제는 보통 본문이 비어 있거나,
204 No Content 형태로 응답됩니다.

즉, “성공적으로 삭제되었다”는 뜻입니다.

### 이 요청에서 확인할 것

* 삭제 후 상세 조회하면 404가 뜨는지
* 목록 조회에서 해당 핑계가 사라졌는지
* HTML 상세 페이지의 삭제 버튼으로도 동작하는지

# 22. HTML 페이지 체크리스트

API만 확인하면 JSON만 보게 되므로,
학생들이 서비스 흐름을 이해하기 어렵습니다.
그래서 아래 페이지도 꼭 같이 확인해 보세요.

## `/`

* 홈 화면이 잘 보이는가?
* 상단 메뉴 링크가 잘 이동하는가?

## `/excuses/`

* “핑계 불러오기” 버튼이 동작하는가?
* API 데이터를 받아 카드 형태로 출력하는가?
* 상세 페이지 링크가 잘 이동하는가?

## `/excuses/<id>/page/`

* 특정 핑계의 상세 내용이 보이는가?
* URL의 id에 따라 다른 데이터가 보이는가?
* 수정 버튼으로 PUT 요청이 되는가?
* 삭제 버튼으로 DELETE 요청이 되는가?

## `/excuses/random-page/`

* 버튼 클릭 시 랜덤 핑계가 나오는가?
* 데이터가 없으면 안내 문구가 보이는가?

## `/excuses/generate-page/`

* situation, tone 입력 후 생성이 되는가?
* 생성 결과가 바로 화면에 표시되는가?
* 생성 후 목록 페이지에서 해당 데이터가 보이는가?
* 결과에 프롬프트 문장이 섞여 나오지 않는가?

# 23. 디버깅 포인트

학생들이 자주 막히는 부분을 미리 정리해 두겠습니다.

## 1. `No module named 'rest_framework'`

원인:

* DRF 설치가 안 되었거나
* 가상환경이 꺼져 있음

해결:

```bash
pip install djangorestframework
```

---

## 2. `No module named 'transformers'`

원인:

* transformers 설치 누락

해결:

```bash
pip install transformers
```

---

## 3. 첫 요청이 너무 느림

원인:

* 모델 다운로드 및 초기 로딩 때문

설명:

* 첫 요청은 오래 걸릴 수 있음
* 이번에는 1.5B 모델이라 처음 로딩이 더 오래 걸릴 수 있음
* 두 번째부터는 비교적 빨라질 수 있음

---

## 4. 랜덤 API에서 오류

원인:

* DB에 데이터가 없는 상태에서 랜덤 선택 시도

해결:

* 먼저 생성 API를 통해 핑계를 1개 이상 저장

---

## 5. CSS가 안 먹음

원인:

* `{% load static %}` 누락
* 정적 파일 경로 오타

해결:

* `base.html` 상단 확인
* `excuses/style.css` 경로 확인

---

## 6. PUT 요청이 안 됨

원인:

* JSON body 누락
* 필드 일부만 보내서 검증 실패

설명:

* `PUT`은 전체 수정 개념이므로 보통 필드를 모두 보내는 것이 안전함

---

## 7. 생성 결과에 영어가 섞임

원인:

* 프롬프트가 영어 기반이거나
* tone 값을 영어와 한국어 혼합으로 넣었거나
* 모델이 프롬프트 일부를 따라쓴 경우

해결:

* 프롬프트를 한국어로 통일
* tone 값을 `진지함`, `가벼움`, `유머`, `죄송함`처럼 통일
* `return_full_text=False` 사용
* 후처리 유지

---

## 8. 생성 결과에 `상황`, `톤`, `조건` 같은 문장이 그대로 출력됨

원인:

* 모델이 프롬프트를 그대로 복사한 경우

해결:

* `return_full_text=False` 사용
* `핑계:` 후처리 적용
* 프롬프트를 짧고 명확하게 유지

---

## 9. favicon 404

브라우저가 자동으로 `/favicon.ico`를 요청하면서 발생하는 경우가 많습니다.
이건 프로젝트 핵심 기능과 무관하므로 무시해도 됩니다.

# 25. 오늘 배운 핵심 정리

이번 실습의 핵심은 아래와 같습니다.

## 1. DRF는 단순 JSON 출력 도구가 아니다.

Serializer, Generic View, URL 설계를 통해 꽤 체계적인 API를 만들 수 있습니다.

## 2. CRUD 위에 AI 기능을 얹을 수 있다.

기존 조회 / 수정 / 삭제 API 위에
“AI 생성 API”를 추가하는 방식은 실제 서비스에서도 자주 쓰이는 아이디어입니다.

## 3. HTML과 API를 같이 보면 이해가 빨라진다.

브라우저에서 눈으로 직접 확인하면,
API가 실제 서비스에서 어떻게 쓰이는지 훨씬 잘 이해할 수 있습니다.

## 4. Service Layer 분리는 중요하다.

AI 로직을 View 안에 전부 넣지 않고 `services.py`로 분리하면,
코드 구조가 훨씬 깔끔해집니다.

# 26. 마무리

이번 교안은 단순 CRUD 실습에서 한 단계 더 나아가,
**DRF + 로컬 LLM + HTML 확인 페이지**를 함께 연결하는 구조를 경험하는 데 의미가 있습니다.

즉, 이번 프로젝트를 통해 아래 흐름을 통째로 익힐 수 있습니다.

```
모델 설계
→ Admin 등록
→ Serializer 작성
→ AI 서비스 함수 연결
→ View 작성
→ URL 연결
→ HTML/CSS 화면 작성
→ 마이그레이션
→ 관리자 계정 생성
→ API 테스트
```

이 구조를 이해하면 이후에 아래와 같은 프로젝트로 쉽게 확장할 수 있습니다.

* AI 짧은 자기소개 생성기
* AI 댓글 생성기
* AI 제목 추천기
* AI 메뉴 추천기

즉, 이번 교안은 단발성 예제가 아니라
앞으로 더 큰 프로젝트로 이어질 수 있는 **기본 뼈대**라고 생각하면 됩니다.
